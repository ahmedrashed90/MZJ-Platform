# خطوات الرفع والتشغيل والـRollback

## قبل الرفع

1. أخذ نسخة احتياطية كاملة من PostgreSQL.
2. حفظ متغيرات البيئة الحالية.
3. التأكد من وجود أسرار Gateway وERPNext وScheduler وTracking.
4. إيقاف عمليات الكتابة أثناء Migration إن كانت البيئة نشطة بكثافة.

## ترتيب الرفع

1. رفع السورس الجديد كاملًا بدل نسخ ملفات متفرقة.
2. تشغيل `database/migrations/20260724_central_access_control_v1190.sql` مرة واحدة.
3. تشغيل `database/seeds/20260724_central_access_catalog.sql` عند الحاجة لإعادة مزامنة الكتالوج؛ الملف Idempotent.
4. نشر الـAPI والواجهة والـWorkers من نفس الإصدار.
5. تسجيل الخروج والدخول بحساب مدير النظام.
6. مراجعة قسم المستخدمين والصلاحيات والسجلات.
7. اختبار مستخدم ممنوع من نظام، ومستخدم محدود الفرع، ومستخدم محدود الإجراء.

## تحقق بعد الرفع

- المستخدم المعطل يحصل على رفض للجلسة.
- النظام غير المفعّل لا يظهر ولا يُستعلم عن بياناته.
- فتح رابط مباشر أو API ممنوع يعيد 403.
- العدادات والتقارير والتصدير تحترم النطاق.
- Tracking Token يعمل، والرابط القديم بـVIN أو رقم الطلب لا يعمل كمعرف عام.
- تعديل صلاحية المستخدم يبطل جلساته أو إصدارها القديم.

## Rollback

الـRollback الصحيح يتكون من جزأين:

1. إعادة نشر سورس الإصدار السابق كاملًا.
2. تشغيل `database/migrations/20260724_central_access_control_v1190_rollback.sql` عند الحاجة لإزالة جداول الربط المركزية الجديدة.

الـRollback لا يحذف البيانات التشغيلية ولا User IDs ولا كلمات المرور، ويترك الأعمدة الإضافية غير المؤذية وTracking Token للحفاظ على البيانات والأمان. استعادة قاعدة البيانات من النسخة الاحتياطية هي الخيار الكامل عند الحاجة للعودة الحرفية.
