# إصلاح ترتيب رسائل الأوتوميشن الواردة — v1.18.0

## المشكلة

عند إرسال العميل رسالتين متتاليتين بسرعة، مثل الاسم ثم رسالة أخرى، كان قفل محادثة الأوتوميشن يبدأ داخل محرك الفلو بعد انتهاء جزء كبير من معالجة التكامل. لذلك كان ممكنًا لطلب الرسالة الثانية أن يصل إلى محرك الفلو قبل الطلب الأول، فتُفحص الرسالة الثانية على الخطوة القديمة.

المثال المرصود:

- سؤال الاسم ظاهر للعميل.
- العميل يرسل `احمد`.
- يرسل بعدها `3` قبل أن يظهر سؤال السيارة.
- الرسالة `3` تُفحص كإجابة للاسم وتظهر رسالة التحقق، بدل تثبيت الاسم والانتقال للسيارة أولًا.

## إعادة البناء

- إضافة قفل تسلسلي على مستوى مدخل التكامل وقبل تهيئة مخطط CRM وحفظ الحدث ومعالجته.
- مفتاح القفل يعتمد على المحادثة الخارجية، أو الصفحة والمشارك، أو رقم الجوال عند غياب معرف المحادثة.
- القفل يُطبق على الرسائل الواردة فقط؛ الرسائل الصادرة وEcho لا تحجز اتصال قفل.
- استخدام Pool مستقل لأقفال PostgreSQL حتى لا يتغير اتصال أو Transaction منطق CRM الحالي.
- الإبقاء على قفل محرك الأوتوميشن الداخلي كحماية إضافية لحالة الجلسة.
- لا توجد Migration جديدة، ولم يتم تعديل الجداول أو تشغيل SQL يدويًا.

## الاختبارات

- نجاح تسلسل حدثين متزامنين لنفس المحادثة: الأول يكتمل قبل بدء الثاني.
- نجاح تنفيذ حدثين لمحادثتين مختلفتين بالتوازي دون قفل عام.
- التأكد أن أعمال CRM تستخدم Pool قاعدة البيانات الأساسي ولا تنتقل إلى Pool الأقفال.
- نجاح فحوصات CRM والأوتوميشن والتراكينج والعمليات وERPNext الموجودة في المشروع.
- نجاح Transpile لجميع ملفات TypeScript وTSX وعددها 128 ملفًا.

## حالة الاعتماد

الإصلاح يحتاج بعد النشر إلى اختبار حي بالترتيب التالي قبل اعتماد النسخة:

1. اختيار مبيعات التمويل.
2. انتظار سؤال الاسم.
3. إرسال اسم صحيح.
4. التأكد أن سؤال السيارة يظهر مرة واحدة.
5. إرسال رسالتين متتاليتين بسرعة والتأكد أن كل رسالة تُفحص على خطوتها الصحيحة.
6. إكمال رقم الجوال والتأكد من إنشاء العميل والتوزيع والرسالة النهائية مرة واحدة.
