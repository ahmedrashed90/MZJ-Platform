# مصادر بيانات داش بورد العمليات

## القاعدة العامة

الداش بورد لا يملك منطق عد منفصلًا عن صفحات العمليات. نقطة التجميع هي:

- `server/_dashboard-data.ts` يستدعي `getOperationsDashboard(user)`.
- `getOperationsDashboard` موجود في `server/_operations-service.ts`.
- الواجهة تقرأ النتيجة من API الداش بورد الحالية، وتفتح تفاصيل المؤشرات عبر API العمليات نفسها.

## المخزون ومؤشرات الفروع

- الأرقام والصفوف التفصيلية تستخدم نفس Scope الخاص بالمستخدم ونفس Status Mapping ونفس شروط الأرشيف والحذف.
- فتح أي مؤشر في الداش بورد يستدعي `listVehicles` مع نفس `metric` والفرع/الموقع.
- التصدير يعيد جميع النتائج المطابقة للفلاتر الحالية، وليس الصفحة الحالية فقط، مع Permission `operations.vehicles.export` وتسجيل Audit.

## الموافقات

- بطاقة الداش بورد وتبويب الموافقات يستخدمان `listApprovals`.
- لا توجد Query بديلة لحساب الموافقات في الواجهة.
- نفس DTO ونطاق الصلاحيات وحالة دورة الموافقات الحالية مستخدمة في الموضعين.

## طلبات النقل والتصوير

- بطاقة الداش بورد وتبويب الطلبات يستخدمان `listRequests`.
- زر التفاصيل في البطاقة يستدعي `getRequestDetail` لعرض سيارات الطلب والمراحل والملاحظات.
- الصادر والوارد والتبويبات تطبق نفس Scope الفروع من السيرفر.

## النواقص

- بطاقة النواقص ونافذة التفاصيل تستخدمان `getShortages`.
- العد والتفاصيل ناتجان من نفس الاستعلام ونفس إزالة التكرار.

### معادلة النواقص

لكل تركيبة فريدة من:

`السيارة + البيان + الموديل + اللون الداخلي + اللون الخارجي`

1. يحسب عدد السيارات الفعلية في الفروع المشمولة.
2. يحسب عدد السيارات المطابقة في المستودع فقط.
3. يضاف للمخزون المستهدف حد أقصى مقداره 3 من سيارات المستودع المطابقة.
4. النقص لكل فرع = `max(0, الهدف - عدد الفرع)` وفق قواعد الفروع المعتمدة.
5. سيارات الوكالة والحالات المستبعدة والإكسسوارات لا تدخل في الحساب.
6. يتم Deduplication بالسيارة/التركيبة قبل تجميع الإجمالي.

اختبار القاعدة موجود في `scripts/test-operations-shortage-rules.mjs`.
