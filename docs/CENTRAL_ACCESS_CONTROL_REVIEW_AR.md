# تقرير مراجعة المستخدمين والصلاحيات قبل التنفيذ

## نطاق المراجعة

تمت مراجعة سورس المنصة الموحدة المرفوع فقط، بما يشمل:

- مسارات React والقائمة الجانبية.
- صفحة الإعدادات الحالية.
- جلسة المستخدم وتسجيل الدخول.
- API المستخدمين.
- جداول `core` الحالية.
- جميع Endpoints الموجودة فعليًا داخل CRM.
- الصفحات الحالية للأنظمة الأربعة.
- شروط الأدوار القديمة المؤثرة في CRM.

## الأنظمة الموجودة فعليًا

| النظام | الحالة داخل السورس المرفوع |
|---|---|
| CRM | موديول فعلي بصفحات وواجهات API وقاعدة بيانات |
| التسويق | صفحة Placeholder فقط داخل المنصة الحالية |
| العمليات | صفحة Placeholder فقط داخل المنصة الحالية |
| التراكينج | صفحة Placeholder فقط داخل المنصة الحالية |

لذلك تم تطبيق حماية النظام والمسار المركزي على الأنظمة الأربعة، وتطبيق حماية الصفحة والإجراء والـAPI بالتفصيل على CRM لأنه الموديول الفعلي الموجود. تم إنشاء كتالوج مركزي لصفحات وإجراءات ومراحل الأنظمة الثلاثة الأخرى دون اختلاق Endpoints أو تغيير فلو غير موجود في السورس المرفوع.

## الوضع السابق

1. كانت صفحة الإعدادات تعرض قسم المستخدمين، لكنها كانت تدير مستخدمًا بدور واحد وفرع واحد وقسم واحد فقط.
2. كان الدخول إلى صفحة الإعدادات وإدارة المستخدمين يعتمد على تحقق إداري عام بدل مفاتيح صلاحيات مستقلة.
3. لم يكن هناك فصل بين دخول النظام، فتح الصفحة، تنفيذ الإجراء، مرحلة سير العمل، ونطاق البيانات.
4. كانت راوتات CRM متاحة للمستخدم المسجل دون Route Guards تفصيلية.
5. عدد من Endpoints كان يعتمد على كون المستخدم مسجلًا أو على دور مدير CRM بدل صلاحية محددة.
6. لم يكن هناك منع فردي يتغلب على الدور، أو سجل منفصل لتغييرات الصلاحيات.
7. صفحة `/activity` كانت Placeholder مستقلًا وليست سجل الأمن المركزي.
8. صفحة CRM القديمة `/crm/admin` كانت مسارًا قديمًا لإعدادات CRM.

## المخاطر التي تمت معالجتها

- حماية الواجهة فقط دون حماية الـBackend.
- منح صلاحيات ضمنية بسبب اسم الدور.
- رؤية مستخدم لبيانات خارج نطاقه بسبب عدم تطبيق Scope على الاستعلام.
- استمرار جلسة مستخدم بعد تعطيله.
- إمكانية منح مدير صلاحيات أعلى من صلاحياته الفعلية.
- حذف إعدادات الصلاحيات عند تعطيل نظام بدل الاحتفاظ بها.
- خلط صلاحيات إعدادات CRM مع إدارة المستخدمين.
- ظهور أزرار التصدير أو الحذف أو النقل دون صلاحية مستقلة.
- استعلام الداش بورد الموحد لجميع الأنظمة قبل التحقق من صلاحية النظام أو نطاق CRM.
- كشف قائمة الأدوار والفروع والأقسام لأي مستخدم نشط عبر API عام للـMetadata.

## الملفات والجداول المستهدفة

### الجداول المطورة أو المضافة

- `core.systems`
- `core.system_pages`
- `core.permissions` (توسعة)
- `core.user_systems`
- `core.user_permission_overrides`
- `core.user_scope_rules`
- `core.permission_change_log`
- `core.users.permission_version`
- توسعات `audit.activity_log`

### الصفحات المركزية

داخل الإعدادات ← المستخدمون والصلاحيات:

1. المستخدمون.
2. الأدوار وقوالب الصلاحيات.
3. الفروع والأقسام.
4. دليل الصلاحيات.
5. سجل تعديلات الصلاحيات.
6. سجل النشاط الأمني.

## قيود المدخلات

الوصف يشير إلى جدول مستخدمين Excel، لكن المرفقات الحالية تحتوي على سورس المنصة وملف النص فقط. لذلك لم يتم إنشاء Seed نهائي بأسماء المستخدمين أو منح صلاحيات لهم بالافتراض. تم تجهيز Seed مركزي للأدوار والأنظمة والصفحات والمفاتيح، ويمكن استيراد جدول المستخدمين لاحقًا بالبريد الإلكتروني مع الحفاظ على User ID وكلمة المرور.
