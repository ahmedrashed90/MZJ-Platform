# تقرير مراجعة نظام الصلاحيات المركزي — MZJ Platform v1.19.1

## مصدر التنفيذ

تمت المراجعة والتنفيذ على السورس النهائي المعتمد:

`MZJ-Platform-main-v1.18.0-MARKETING-PHOTO-REQUEST-FOLLOWUP-CLEAN-20260724(1).zip`

سورسات CRM والتسويق والتراكينج والعمليات القديمة استُخدمت لفهم الوظائف ودورات العمل فقط، ولم يُنسخ منها كود أو قواعد Firebase أو صفحات مستخدمين إلى المنصة.

## النتيجة العامة

المنصة تحتوي الأنظمة الأربعة فعليًا. كان نظام الصلاحيات الحالي يوفر جداول أساسية للأدوار والصلاحيات والفروع والأقسام، لكنه لم يكن يغطي بصورة موحدة:

- تفعيل النظام لكل مستخدم.
- نطاقًا مستقلًا لكل نظام.
- صلاحيات الصفحات والإجراءات ومراحل العمل.
- السماح والمنع الفردي.
- إبطال الجلسات عند تعديل الصلاحيات.
- سجل تعديلات الصلاحيات وسجل النشاط الأمني المفصل.
- منع الاستعلام عن بيانات نظام غير مسموح بدل جلبها ثم إخفائها.

## ما تم الحفاظ عليه دون تغيير

- حالات العملاء وتوزيعهم داخل CRM.
- مراحل العمليات وترتيبها وشروطها التشغيلية.
- مراحل التراكينج العشر وترتيبها.
- فلو الحملات والأجندات وTask Template وإجراءات التكليف والنشر.
- الحسابات والنسب والتقارير التشغيلية.
- User IDs وكلمات المرور والبيانات التشغيلية الحالية.
- تصميم الأنظمة، باستثناء واجهة الإدارة المركزية وحالات منع الوصول.

## أبرز المشكلات التي عولجت

1. استنتاج دخول النظام من الدور أو القسم بدل سجل صريح لكل مستخدم ونظام.
2. استخدام دور إداري عام بدل مفتاح صلاحية دقيق في بعض المسارات.
3. تكرار إدارة إعدادات أو فروع داخل الأنظمة بدل المصدر المركزي.
4. صلاحيات واجهة لا يكفي وجودها دون تحقق Backend ونطاق بيانات.
5. رابط التتبع العام القابل للوصول برقم الطلب أو VIN.
6. عدم فصل صلاحية مشاهدة فحص أخطاء CRM عن تنفيذ التصحيح.
7. عدم فصل إدارة توزيع CRM والأتمتة عن مجرد مشاهدة إعدادات CRM.
8. احتمال استمرار جلسة قديمة بعد تعديل صلاحيات المستخدم أو تعطيله.

## قاعدة التنفيذ

طبقة الصلاحيات تحيط بالفلو الحالي ولا تعيد تصميمه. نقل المستخدم بين فرع أو قسم يغيّر نطاق وصوله فقط، ولا ينقل العملاء أو الطلبات أو التاسكات تلقائيًا.
