# تقرير تنفيذ إعدادات أوتوميشن عملاء CRM — الإصدار 1.18.0

## النتيجة

تم تنفيذ تبويب مستقل باسم **إعدادات الأوتوميشن** داخل:

`الإعدادات → إعدادات CRM → إعدادات الأوتوميشن`

أصبحت هذه الإعدادات المصدر التشغيلي المركزي لرسائل دخول العميل، وربط المنصات والـWorkers، وسياسة التشغيل، واختيارات الخدمات، وخطوات الفلو، والتحقق من الإجابات، ومدة الانتظار، وكلمات الإلغاء وإعادة البداية.

تم تنظيف تبويب **دخول وتوزيع العملاء** من تعديل رسائل واختيارات الخدمة المكرر، وأصبح يوضح حدود الربط بين الأوتوميشن ومحرك توزيع الموظفين، مع روابط مباشرة إلى كل إعداداتهما.

## الفصل بين الأوتوميشن والتوزيع

- الأوتوميشن يحدد الخدمة والقسم والفرع، وينشئ أو يعيد استخدام طلب الخدمة، ثم يكمل الأسئلة ورسالة النهاية.
- محرك التوزيع يحاول إسناد مندوب المبيعات والكول سنتر وفق المنطق الأصلي.
- عدم وجود موظف متاح أو حدوث خطأ داخل محاولة التوزيع لا يوقف فلو الأوتوميشن، ولا يعيد رسالة البداية، ولا يرسل رسالة من محرك التوزيع.
- في حالة تعذر الإسناد، يبقى الطلب بحالة توزيع `pending` مع تسجيل سبب الخطأ داخل Metadata الطلب وسجل الملكية.
- لم تتم إعادة كتابة خوارزمية التوزيع العامة، ولم يتم تغيير قواعد الدور أو استقبال العملاء خارج نقطة الربط اللازمة للأوتوميشن.

## بنية التخزين

تم توسيع السجل المركزي `crm.automation_settings` لحفظ:

- حالة واسم الأوتوميشن.
- ربط المنصات بالـWorkers.
- سياسة وتوقيت التشغيل.
- الرسائل العامة.
- اختيارات الخدمات وترتيبها وحالتها.
- فلو وخطوات كل اختيار.
- مدة انتظار الإجابة.
- كلمات الإلغاء وإعادة البداية.
- رقم نسخة الإعدادات وآخر تعديل والمستخدم المعدّل.

تم إنشاء `crm.customer_automation_runs` لحفظ حالة كل تشغيل، ومنها:

- العميل والمحادثة والمنصة والـWorker.
- اختيار الخدمة والخطوة الحالية والمحاولات.
- الإجابات وسجل الانتقالات.
- آخر Event/Message ورسالة أوتوماتيكية.
- حالة وسبب انتهاء الفلو.
- نسخة كاملة من الإعدادات وقت بداية الفلو؛ لذلك تعديل الإدارة لا يغيّر فلو عميل بدأ بالفعل.

تمت إضافة قيود تمنع وجود أكثر من فلو نشط للمحادثة، وكذلك أكثر من فلو نشط لنفس العميل عبر محادثات أو منصات مختلفة.

## منع التكرار والتزامن

- يتم استخدام بصمة ثابتة للرسالة الواردة في `event_key` لمنع معالجة Webhook المكرر.
- حجز حدث الأوتوميشن أصبح Atomic قبل التنفيذ.
- يوجد قفل قصير على مستوى العميل أثناء تحديد الانتقال التالي في الـState Machine.
- الرسائل التي تصل أثناء إرسال سؤال أو إنشاء الطلب تتحول إلى `deferred`، ثم تُعالج بالترتيب بعد استقرار حالة الفلو، بدون `setTimeout` عشوائي.
- رسائل الإرسال تستخدم Idempotency Key مرتبطًا بالتشغيل والمرحلة لمنع إرسال نفس الرسالة مرتين.
- الأرقام المرسلة لاختيار الخدمة تُطابق رقم الاختيار كاملًا؛ مثلًا `10` لا يطابق الاختيار `1`.

## ربط المنصات والـWorkers

قائمة الربط تُبنى من `crm.integration_endpoints` الموجودة فعليًا في المشروع، ولا تسمح بحفظ Worker غير موجود أو غير تابع للمنصة. تم دعم الربط الحالي لـFacebook وInstagram وWhatsApp وTikTok وSnapchat وأي Endpoint فعلي يظهر من قاعدة البيانات.

تقرأ نقاط استقبال الرسائل الإعدادات من Service مركزي مع Cache قصير لمدة 30 ثانية، ويتم مسح الـCache مباشرة بعد الحفظ.

## الصلاحيات

قراءة وتعديل إعدادات الأوتوميشن محميان في الـBackend والواجهة بصلاحية إدارة CRM الحالية عبر `isCrmManager`. الـWorkers تقرأ النسخة التشغيلية فقط ولا تمتلك Endpoint لتعديل الإعدادات.

## نتائج الفحص

نجحت الفحوصات التالية:

- جميع ملفات `scripts/check-*.mjs` الموجودة بالمشروع: **23 فحصًا ناجحًا**.
- فحص الأوتوميشن الجديد، بما يشمل المصدر المركزي، الـState Machine، منع التكرار، وحماية استقلال التوزيع.
- فحص تحويل TypeScript/TSX للملفات المعدلة: **12 ملفًا ناجحًا**.
- مراجعة Imports ومسار API والمهاجرات الجديدة وعدم وجود Worker مكرر أو Route موازٍ للأوتوميشن القديم.

تمت محاولة تشغيل `tsc -b` الكامل، لكنه لم يكتمل في بيئة العمل لأن حزم المشروع مثل React وVite وملفات أنواعها غير مثبتة، ومحاولة جلبها تعذرت بسبب عدم توفر اتصال بسجل الحزم. لذلك لا يتم الادعاء بأن Build إنتاجي كامل تم تشغيله داخل هذه البيئة. يلزم بعد فك الملف تشغيل `pnpm install` ثم `pnpm run build` في بيئة النشر المعتادة.

لم تتوفر في بيئة العمل قاعدة بيانات تشغيلية أو مفاتيح Workers حقيقية، لذلك لم يتم إرسال رسائل حقيقية للعملاء. الفحوصات المنفذة هي فحوصات بنية ومنطق وتحويل للكود.

## تأكيد نطاق التعديل

لم يتم تعديل الداش بورد أو التقارير أو صفحة المحادثات أو إضافة العملاء أو العمليات أو التراكينج أو التسويق أو تسجيل الدخول أو القائمة الجانبية. التعديلات خارج صفحة الإعدادات اقتصرت على الربط التقني الضروري لمحرك الأوتوميشن مع استقبال الرسائل، إنشاء طلب الخدمة، ومنع تعارضه مع التوزيع الحالي.
