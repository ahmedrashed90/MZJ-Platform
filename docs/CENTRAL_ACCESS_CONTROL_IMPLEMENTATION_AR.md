# شرح تنفيذ نظام الصلاحيات المركزي

## المصدر الواحد

صفحة **الإعدادات ← المستخدمون والصلاحيات** هي المصدر الوحيد لإدارة:

- المستخدمين.
- الأدوار وقوالب الصلاحيات.
- الفروع والأقسام.
- الأنظمة المسموحة.
- نطاق البيانات لكل نظام.
- الصفحات والإجراءات ومراحل العمل.
- السماح والمنع الفردي.
- سجل تعديلات الصلاحيات.
- سجل النشاط الأمني.

المسارات القديمة الخاصة بإدارة المستخدمين أو الإعدادات المكررة تُعاد توجيهها إلى الإعدادات المركزية أو تُوقف كتابتها.

## حساب الصلاحية النهائية

الترتيب المطبق:

1. المستخدم المعطل ممنوع، وتُرفض جلسته القديمة.
2. المنع الفردي الصريح يتغلب على كل السماحات.
3. السماح الفردي الصريح.
4. صلاحيات الدور أو الأدوار العامة.
5. قالب الدور المحدد داخل النظام.
6. المنع الافتراضي عند عدم وجود صلاحية.

صلاحية `platform.superadmin` لا تتجاوز منعًا فرديًا صريحًا.

## مستويات التحقق

كل طلب محمي يمر عبر:

1. التحقق من الجلسة وحالة المستخدم وإصدار الصلاحيات.
2. التأكد من تفعيل النظام للمستخدم.
3. التحقق من صلاحية الصفحة أو الإجراء أو المرحلة.
4. تطبيق نطاق البيانات في استعلام السيرفر.
5. التحقق من حالة الفلو الحالية وشروطه الأصلية.
6. تنفيذ العملية.
7. تسجيل النجاح أو الرفض أو الفشل في Audit Log.

## تعطيل نظام لمستخدم

تعطيل النظام:

- يخفي النظام وصفحاته وإجراءاته من الواجهة.
- يمنع الراوت والـAPI والبيانات من السيرفر.
- لا يحذف الدور أو الفروع أو الأقسام أو الاستثناءات المخزنة.
- يسمح بإعادة التفعيل واستعادة الإعدادات السابقة.

## نقل المستخدم

تعديل الفرع أو القسم يحدث نفس حساب المستخدم ويحافظ على User ID والسجل وكلمة المرور. لا تتم إعادة إسناد العملاء أو الطلبات أو التاسكات تلقائيًا؛ أي إعادة توزيع تظل إجراءً منفصلًا حسب فلو النظام الحالي.

## إبطال الكاش والجلسات

كل تعديل مؤثر يزيد `permission_version` ويحذف جلسات المستخدم عند الحاجة، فلا تستمر صلاحية قديمة بعد التعطيل أو النقل أو تغيير الدور والنطاق.

## عدم تعديل الفلو

لم تتغير أسماء الحالات أو المراحل أو ترتيبها أو خوارزميات التوزيع أو حسابات التقارير. تمت إضافة بوابات تحقق فقط قبل القراءة أو التنفيذ.
