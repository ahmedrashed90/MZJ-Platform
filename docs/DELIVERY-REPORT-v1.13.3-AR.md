# تقرير تسليم MZJ Platform v1.13.3 — Operations Native

## النتيجة

تمت إعادة بناء وحدة العمليات Native داخل المنصة الأساسية بعد Tracking، باستخدام React وTypeScript وAPI المنصة وPostgreSQL والجلسة والصلاحيات المركزية. لم يتم استخدام محاولة عمليات سابقة كأساس، ولم يتم نسخ كود النظام القديم.

## أهم الأسباب الجذرية التي عولجت

1. **صفحة العمليات كانت Placeholder:** لم توجد صفحات أو APIs أو Domain logic فعلية، فتم إنشاء الوحدة داخل Architecture المنصة بدل تركيب HTML قديم.
2. **الصلاحيات كانت Roles فقط:** تم تحميل Permission codes داخل Session وإنشاء Guards مركزية وفحص Branch/Location scope في Backend.
3. **رسالة PostgreSQL الخاطئة:** كانت استعلامات الداش بورد داخل Failure واحد؛ فشل Query واحد كان يخفي كل الأقسام. أصبحت Widgets معزولة مع Connection check حقيقي.
4. **الحركة والطلبات كانت معرضة للسجلات اليتيمة:** تم ترتيب الإدخالات واستخدام Transactions وقفل السجلات وRollback.
5. **سجل الحركة كان معرضًا لاختلاف أسماء الأعمدة:** الاستعلامات الجديدة مبنية على Schema الفعلية وتستخدم `created_at` بصورة موحدة.
6. **الموافقات لم تكن مرتبطة بدورة تسليم:** تم إنشاء Approval cycle لكل دخول جديد إلى `مباع تحت التسليم` مع تاريخ غير قابل للمسح.
7. **Tracking كان يعتبر رقم الطلب الهوية الوحيدة:** تم فصل Business order number عن Source identity، وإلغاء الحظر الدائم لرقم الطلب بعد الحذف.
8. **الحذف كان قد يمس التاريخ عبر Cascade:** تم إنشاء فحص علاقات صريح وسجل Audit مستقل، وتقييد العلاقات التاريخية الحساسة.
9. **استيراد Excel كان قادرًا نظريًا على تجاوز الفلو:** تم منع الحالات التشغيلية الحساسة من الشيت وإضافة Preview وصلاحية منفصلة للاستبدال الكامل.
10. **جدول المخزون لم يحقق سلوك الجدول الاحترافي:** تم تنفيذ العرض الكامل، Sticky header، تمرير متزامن، Resize، Auto-fit، Reset، وحفظ تفضيلات العرض فقط.
11. **تفاصيل النقل كانت غير كافية للتدقيق:** أضيفت نافذة تفاصيل كاملة للطلب والسيارات والمراحل والأحداث مع البحث والتصدير.
12. **المواقع كانت مرتبطة بفرع واحد فقط:** أضيف ربط متعدد الفروع مع فرع أساسي للتوافق، وإدارة الربط من إعدادات العمليات.
13. **إجراءات الموافقات والحركة كانت معرضة لتنفيذ غير مقصود:** استبدلت رسائل المتصفح بنوافذ المنصة وأضيفت مراجعة نهائية للحركة وأسباب إلزامية للتراجع والمسح.

## ما لم يتم ادعاء اختباره

لم تتوفر في بيئة التنفيذ قيمة `DATABASE_URL` لقاعدة PostgreSQL الفعلية أو نسخة Staging منها. لذلك لم يتم:

- تشغيل المهاجرة على قاعدة Production.
- فحص أعداد VINs والبيانات الفعلية الموجودة على السيرفر.
- اختبار صلاحيات المستخدمين الحقيقيين والفروع الفعلية على بيانات Production.
- اختبار Google Sheet وNEXT ERP حيًا من الشبكة.
- اختبار Concurrency متعدد الأجهزة ضد قاعدة خارجية فعلية.

تم إرفاق `scripts/test-operations-postgres.mjs` لتشغيل Smoke test على `TEST_DATABASE_URL` منفصلة قبل Main.

## خطة النشر الآمنة

1. إنشاء Backup كامل:

```bash
pg_dump --format=custom --file=mzj-before-operations-v1.13.3.dump "$DATABASE_URL"
```

2. إنشاء نسخة Staging من قاعدة البيانات، وتشغيل:

```bash
psql "$TEST_DATABASE_URL" -v ON_ERROR_STOP=1 -f database/migrations/20260720_operations_native.sql
TEST_DATABASE_URL="$TEST_DATABASE_URL" pnpm run test:operations:postgres
```

3. مراجعة:

- عدد السيارات وVINs قبل وبعد.
- عدم وجود VIN مكرر.
- ربط المواقع بالفروع في `operations.location_branches`.
- ظهور الصلاحيات الجديدة للأدوار الصحيحة.
- ربط طلبات Tracking القديمة بالـVIN بصورة فريدة قبل Backfill لـ`operations_vehicle_id`.

4. تشغيل Build النهائي ثم النشر.

5. بعد النشر اختبار فعلي محدود:

- سيارة اختبار بلا تاريخ.
- حركة فردية وجماعية.
- طلب نقل كامل المراحل.
- دورة موافقات كاملة.
- حذف سيارة بلا تاريخ ثم مراجعة Audit.
- حذف طلب Tracking تجريبي والتحقق من بقاء سيارة العمليات.

## Rollback

المهاجرة Additive في الأساس ولا تقوم Drop للجداول التاريخية. لذلك Rollback الآمن هو:

1. إرجاع نسخة الكود السابقة إذا ظهر خلل في الواجهة أو API.
2. عدم حذف الجداول الجديدة أو البيانات التاريخية يدويًا.
3. عند وجود مشكلة بيانات حقيقية، استعادة Backup إلى قاعدة منفصلة أولًا ومقارنة الفروقات.
4. لا يتم تشغيل SQL عكسي تدميري على Main دون مراجعة العلاقات والبيانات التي أُنشئت بعد النشر.

## ملاحظة Dependency Audit

أظهر `npm install` تحذيرات Audit في شجرة الاعتماديات الحالية. لم يتم استخدام `npm audit fix --force` لأن ذلك قد يغيّر إصدارات رئيسية ويؤثر على أنظمة خارج نطاق المهمة. يجب معالجة Audit في مهمة مستقلة مع Regression testing شامل.
