# تقرير تنفيذ نظام العمليات Native V2

## المصدر المعتمد

تم تنفيذ الوحدة داخل نسخة كاملة من:

`MZJ-Platform-v1.13.2-Tracking- FULL.zip`

استخدم سورس العمليات القديم `MZJ-Operations-Legacy-Final-Reference.zip` كمرجع لفهم الفلو والمسميات والحالات فقط. لم يتم نسخ HTML أو JavaScript أو CSS أو Firebase أو Firestore من النظام القديم.

إصدار المنصة في هذه النسخة: `1.14.0`.

## ما تم تنفيذه

- نظام عمليات Native داخل React وTypeScript وAPI المنصة وPostgreSQL.
- تبويبات موحدة: مخزون السيارات، إدارة السيارات، الحركة، طلبات النقل، الموافقات، جميع السيارات، سجل الحركات، الأرشيف.
- مخزون كامل ببحث وفلاتر وPagination من السيرفر وتصدير كل النتائج المطابقة.
- جدول ممتد بعرض الكارت مع تغيير عرض الأعمدة بالسحب، ضبط تلقائي بالضغط المزدوج، إعادة ضبط المقاسات، وحفظ المقاسات كتفضيل واجهة فقط.
- إضافة/تعديل/استيراد السيارات بثلاثة أوضاع: استبدال تشغيلي آمن، إضافة الجديد فقط، تحديث الموجود فقط.
- تصدير ملفات Excel حقيقية بصيغة `.xlsx` وقراءة `.xlsx` وCSV والقوالب القديمة، مع حفظ VIN كنص وعدم فقد الأصفار في بدايته.
- عزل أخطاء صفوف الاستيراد داخل PostgreSQL Savepoints حتى لا يؤدي خطأ صف واحد إلى إفساد Transaction الدفعة كلها.
- حركة سيارة واحدة أو عدة سيارات داخل Transaction واحدة، وسجل مستقل لكل سيارة وBatch مشترك.
- تشيك مستقل لكل سيارة عند وجودها في الوكالة.
- فلو مباع تحت التسليم ودورة موافقات مالية وإدارية مستقلة، ومنع مباع تم التسليم قبل اكتمالهما.
- طلبات نقل متعددة السيارات ومراحل مرتبة وصلاحيات الفرع المصدر والمستهدف.
- تحديث موقع السيارات وإنشاء الحركة وتحديث مرحلة الاستلام داخل Transaction واحدة.
- مسح السيارة الفيزيائي فقط عندما لا يوجد أي تاريخ تشغيلي، مع Snapshot تدقيق دائم.
- أرشفة منطقية منفصلة عن الحذف الفيزيائي.
- تكامل Tracking مجمع داخل المخزون بدون API لكل صف.
- مسح طلب التراكينج باستخدام internal ID وTransaction واحدة مع Snapshot دائم وفك ارتباط السيارات دون حذفها.
- منع رجوع نسخة المصدر القديمة المحذوفة باستخدام `source_identity`، مع السماح بطلب جديد يحمل نفس رقم الطلب من هوية مصدر جديدة.
- عدم استخدام رقم الطلب كـUnique identity أو قائمة حظر دائمة.
- Modal Stack مركزي يستخدم Portal، ونافذة تأكيد المسح تظهر فوق نافذة التفاصيل.
- كروت فروع الداش بورد تفتح نوافذ بحث وتصدير Excel بنفس Query ونطاق الصلاحيات المستخدمين في العد.
- كارت طلبات النقل والتصوير يفتح تبويبي النقل والتصوير ويعرض المنشئ والتاريخ وجميع سيارات الطلب.
- فشل Widget واحد في الداش بورد لا يعرض رسالة أن PostgreSQL غير مربوط، ولا يخفي بقية الأقسام الناجحة.
- دور `system_admin` وصلاحيات العمليات ومسح طلبات التراكينج داخل نظام الصلاحيات المركزي.
- فحص نطاق الفرع داخل Backend في الإضافة والتعديل والمسح والأرشفة والحركة والموافقات والاستيراد؛ والاستبدال الكامل لا يعطل سيارات فروع أخرى.
- تسجيل Event Outbox دائم لكل سيارة عند مسح طلب التراكينج، حتى يظل تاريخ ارتباطها محفوظًا ويمنع حذفها الفيزيائي لاحقًا باعتبارها سيارة بلا تاريخ.
- إعدادات المواقع والحالات داخل صفحة إعدادات المنصة.

## السبب الجذري للمشكلات السابقة

### خطأ الحركات وطلبات النقل

المحاولات السابقة كانت معرضة لإنشاء سجل تابع قبل السجل الأساسي أو تنفيذ تحديثات متفرقة خارج Transaction واحدة. التنفيذ الحالي ينشئ الحركة الأساسية أولًا، ثم يحدث السيارة والسجلات التابعة، ويعمل Rollback كامل عند فشل أي عنصر.

### رسالة PostgreSQL غير مربوط

كانت الداش بورد تربط نجاح الاتصال بنجاح كل استعلامات الأقسام. أصبح اختبار الاتصال `select 1` مستقلًا، وكل قسم يعيد حالته بصورة مستقلة.

### خطأ مسح طلب التراكينج

تمت إزالة الاعتماد على رقم الطلب كهوية للحذف أو منع التكرار. الحذف الحالي:

1. يقفل الطلب باستخدام internal ID و`FOR UPDATE`.
2. يجلب الطلب والسيارات والمراحل والأحداث ورسائل SMS.
3. يحفظ Snapshot دائمًا في `tracking.deleted_orders`.
4. يحذف الطلب داخل Transaction واحدة؛ علاقات المراحل والسيارات تُزال بقواعد العلاقات، ورسائل SMS تحتفظ بتاريخها بعد فك الروابط.
5. لا يكتب في `tracking.deleted_order_blocks`.
6. يعيد `requestId` آمنًا عند الخطأ، مع تسجيل التفاصيل التقنية في السيرفر.

رقم المرجع `ops-mrsg1s3c-de275f7c` لا يمكن تتبعه داخل هذه البيئة لأن Logs بيئة الإنتاج غير متاحة، لذلك تم إصلاح المسار بالكامل من الـpayload حتى SQL وForeign Keys بدل تغيير الرسالة فقط.

### نافذة المسح خلف Popup

تم إنشاء `Modal` مركزي يستخدم `createPortal(document.body)` مع مستويات طبقات موحدة، وإغلاق أعلى نافذة فقط بزر Escape، وقفل تمرير الخلفية، وإعادة التركيز لزر المسح عند الإلغاء.

## مصدر الحقيقة

- السيارة وحالتها وموقعها: `operations.vehicles` مع `operations.locations` و`operations.vehicle_statuses`.
- الحركة: `operations.movements` و`operations.movement_batches`.
- طلبات النقل: `operations.transfer_requests` و`operations.transfer_request_vehicles` و`operations.transfer_request_events`.
- الموافقات الحالية: أحدث سجل نشط في `operations.vehicle_approvals`.
- تاريخ الموافقات: `operations.approval_events`.
- Tracking: `tracking.orders` و`tracking.order_vehicles` و`tracking.vehicle_stages`.
- عمود Tracking في المخزون: Query مجمعة تربط `tracking.order_vehicles.vehicle_id` مع VIN كمرجع احتياطي.
- أرقام كروت الداش بورد والقوائم التفصيلية: نفس تعريفات الحالات ونطاق الفروع؛ Endpoint التفاصيل يعيد `total` من نفس شروط القائمة.

## Migration

الملف:

`database/migrations/20260720_operations_native_v2.sql`

خصائصه:

- غير تدميري للتاريخ.
- Idempotent قدر الإمكان.
- لا يحذف السيارات أو الحركات أو الطلبات أو الموافقات.
- يزيل فقط قيد Unique الخاطئ عن `sales_order_no` ويستبدله بهوية مصدر.
- يعطل أي حظر قديم دائم لأرقام الطلبات دون حذف سجل التدقيق.

ملف إرشاد التراجع:

`database/migrations/20260720_operations_native_v2_rollback.sql`

لا يوجد Rollback تدميري يحذف الجداول أو التاريخ. التراجع الآمن يعتمد على Snapshot قاعدة البيانات قبل التشغيل.

## الاختبارات التي تم تشغيلها فعليًا

نجحت:

- API import extension check.
- جميع اختبارات CRM البنيوية الموجودة في السورس.
- Tracking native module check.
- Operations Native V2 structure and safety check.
- فحص Transpile نحوي لكل ملفات TS/TSX في `src` و`server` و`api`: 109 ملفات، صفر أخطاء نحوية.
- فحص TypeScript مستقل وصارم لوحدة Excel الجديدة: نجح بدون أخطاء.
- إنشاء ملف XLSX فعلي واختبار سلامة ZIP الداخلي: نجح، وتم التأكد من بقاء VIN تجريبي يبدأ بـ`000123`.
- مطابقة نص Migration مع Schema Runtime: 371 سطرًا متطابقًا.
- فحص ZIP النهائي واختبار سلامة الأرشيف يتم قبل التسليم، وتُسجل نتيجته في ملف نتائج الاختبارات.

لم يمكن تشغيله داخل بيئة التنفيذ الحالية:

- `pnpm install` وProduction Build: Corepack حاول الوصول إلى `registry.npmjs.org` وفشل بـ`EAI_AGAIN` لأن الشبكة غير متاحة.
- TypeScript project check الحقيقي: Dependencies وType declarations غير مثبتة بسبب المشكلة السابقة. تم تشغيل TypeScript Transpile وفحوص بنيوية بدلًا منه، ولا يُعتبر ذلك بديلًا مساويًا للـBuild.
- اختبارات PostgreSQL الفعلية: لا يوجد `DATABASE_URL` ولا خادم/أداة PostgreSQL داخل البيئة، لذلك لم يتم تشغيل Migration أو حركة أو طلب نقل أو مسح طلب على قاعدة حقيقية.
- Logs الإنتاج الخاصة برقم المرجع السابق غير متاحة.

لا يتم الادعاء بأن هذه الاختبارات غير المنفذة نجحت.
