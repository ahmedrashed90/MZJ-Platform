# تقرير تنفيذ إعادة بناء نظام التسويق Native

## مصدر التنفيذ

- المصدر الأساسي المعدل: سورس منصة MZJ الموحدة `v1.18.0` المرفوع.
- سورس التسويق القديم استُخدم كمرجع للفلو والحقول والتصميم فقط.
- لم يتم تضمين سورس التسويق القديم أو تشغيله داخل `iframe`، ولم يُستخدم Firebase كـRuntime.

## النتيجة المعمارية

تم إنشاء نظام تسويق Native داخل المنصة باستخدام:

- React وTypeScript لواجهات المستخدم.
- API المنصة الموحدة تحت `/api/marketing`.
- PostgreSQL داخل مخطط `marketing` مع ربط منظم بمخطط `operations` لطلبات التصوير والاستوك.
- تسجيل الدخول والمستخدمين والصلاحيات الحالية في المنصة.
- الإعدادات من صفحة الإعدادات الموحدة، دون Route مستقل لإعدادات التسويق.

## الصفحات المنفذة

1. لوحة التحكم.
2. قاعدة بيانات التسويق.
3. إنشاء حملة بخمس خطوات.
4. إنشاء أجندة بثلاث خطوات.
5. إدارة الحملات والأجندات.
6. إدارة الباقات.
7. تجهيز النشر.
8. متابعة طلبات التصوير.
9. تقويم التسويق.
10. استوك العمليات للقراءة فقط.
11. تقارير التسويق.
12. الحضور والانصراف.
13. ربط المنصات.

كما تمت إضافة صفحة طلبات التصوير داخل العمليات لقراءة وتحديث نفس السجل المشترك.

## نموذج الحملات والأجندات

- الحملة أو الأجندة هي سجل واحد في `marketing.campaigns`.
- كل نسخة كرييتيف هي `Creative Instance` مستقلة في `marketing.creatives`، وتحمل رقمًا مثل `N01` و`N02`.
- السيارات مرتبطة بالـInstance عبر `marketing.instance_vehicles` وتُقرأ من استوك العمليات.
- الميزانية وجدول النشر مرتبطان بالـInstance، لا باسم الكرييتيف فقط.
- الحفظ يتم داخل Transaction واحدة لمنع البيانات الجزئية.
- `idempotency_key` ثابت داخل مسودة المتصفح ويمنع تكرار الحملة والتاسكات عند إعادة الضغط.
- أرقام التاسكات تُنشأ من Sequence في السيرفر مع Unique Constraint في قاعدة البيانات.

## علاقة Task Template والتاسكات التنفيذية

- يتم إنشاء Task Template مستقلة لكل: `Creative Instance × كاتب محتوى`.
- يتم إنشاء Execution Task مستقلة لكل: `Creative Instance × قسم × يوزر تنفيذي × كاتب محتوى`.
- كل Execution Task تحمل `template_task_id` الخاصة بالكاتب والـInstance نفسهما.
- اعتماد Template يفتح التاسكات المرتبطة بها فقط.
- لا يمكن استلام أو تنفيذ Execution Task قبل اعتماد Template المرتبطة.
- تاريخ الاستلام الفعلي يُسجل مرة واحدة عند الضغط على «تم الاستلام» ولا يتغير مع إعادة الضغط.
- الملف النهائي محفوظ تحت رقم التاسك الفعلي، لذلك لا يحدث دمج بسبب تشابه اسم الكرييتيف.

## تقدم التاسكات والأقسام

- إجراءات التكليف ونسبها تُقرأ من `marketing.assignment_actions`.
- إجراءات الإدارة فقط محمية في الاستجابة وفي تنفيذ الـAPI، وليست مخفية شكليًا فقط.
- تقدم التاسك يُحسب من نسب الإجراءات المطلوبة المكتملة، مع تطبيع المجموع إذا لم يكن 100.
- تقدم القسم هو متوسط تاسكاته التنفيذية.
- تقدم الحملة هو متوسط الأقسام بالتساوي.
- النقل إلى قسم النشر محمي في السيرفر ولا يتم قبل وصول الأقسام إلى 100%.

## إعدادات التسويق الموحدة

أضيفت إدارة البيانات التالية داخل صفحة إعدادات المنصة:

- الأقسام وربط يوزرات كل قسم وتحديد قسم المحتوى.
- أنواع الكرييتيف والكود المختصر والقسم الأساسي.
- إجراءات التكليف والنسبة والجمهور: مستخدم أو إدارة أو كليهما.
- أنواع الحملات وأكوادها وتسلسلها.
- المنصات وأنواع النشر والأبعاد.
- تصنيفات الباقات.
- حالات طلبات التصوير.
- مواعيد الحضور والانصراف وفترة السماح.

جميع القوائم الأساسية في التسويق تُقرأ من قاعدة البيانات، وليست Constants داخل الصفحات.

## طلبات التصوير المشتركة

- الاستوك في التسويق يقرأ `operations.vehicles` فقط.
- طلب التصوير يُنشأ في `operations.photography_requests`.
- السيارات المرتبطة بالطلب محفوظة في `operations.photography_request_vehicles`.
- التسويق والعمليات يقرآن السجل نفسه.
- الحالات تُقرأ من `marketing.request_statuses` في النظامين.
- كل تحديث حالة يُسجل في `operations.photography_request_events` ويظهر كسجل متابعة في الصفحتين.
- منع إنشاء طلب تصوير نشط ثانٍ للسيارة نفسها.

## الملفات والروابط والتصدير

- رفع ملفات Task Template وإصدارات التعديل والملفات النهائية.
- فصل صلاحية رفع Template عن صلاحية رفع الملف النهائي.
- رفع ملفات المشروع والنتائج وربط روابط المنصات.
- تنزيل Task Template كملف Excel متوافق.
- تصدير مراجعة المشروع إلى Excel متوافق، ويشمل التاسكات والتكليفات والميزانية وجدول النشر.
- طباعة الصفحات والتقارير عبر متصفح المنصة إلى PDF.

## الحضور وربط المنصات

- الحضور والانصراف يستخدم مستخدم المنصة الحالي دون Login منفصل.
- تسجيل الحضور والتأخير والانصراف ودقائق العمل في PostgreSQL.
- Presence heartbeat من صفحات التسويق.
- إعدادات الدوام قابلة للتعديل من إعدادات التسويق.
- صفحة ربط المنصات تحفظ Metadata وحالة الاتصال فقط، ولا تعرض أو تحفظ Secrets داخل الواجهة.

## الأجزاء المستبعدة عمدًا

لم يتم إنشاء أو نقل:

- Checklist ريل السيارات.
- Local Publisher.
- Publisher Agent أو Electron.
- iframe للنظام القديم.
- Firebase Runtime.
- صفحة إعدادات مستقلة داخل التسويق.

## حالة التحقق النهائية

فحوص البنية والسيناريوهات وصياغة TypeScript وفحوص المنصة الثابتة نجحت. تعذر تثبيت الاعتماديات وتشغيل Build كامل داخل بيئة التنفيذ بسبب فشل DNS إلى `registry.npmjs.org`. لذلك لا يُعتبر هذا التقرير إثبات Build إنتاجي، ويجب تنفيذ خطوات التحقق الإلزامية في ملف النشر قبل اعتماد النسخة للإنتاج.
