# تسليم إعادة بناء نظام العمليات داخل المنصة الموحدة

## مصدر التنفيذ
- تم البناء داخل سورس المنصة الموحدة الحالي باستخدام React وTypeScript وواجهات API الحالية وPostgreSQL.
- استُخدم سورس العمليات القديم لفهم الفلو والمسميات والشروط فقط، بدون نسخ HTML أو JavaScript أو CSS أو Firebase منه.
- أزيل التنفيذ الأحادي والمبسّط لوحدة العمليات، وأعيد تقسيمه إلى صفحات وخدمات API ومكوّنات مستقلة قابلة للصيانة.
- لم تُنشأ لوحة عمليات منفصلة؛ استمرت الداش بورد الموحدة في القراءة من جداول العمليات الجديدة.
- صفحات Tracking المستقلة وCRM والتسويق لم تُحذف، والتعديلات المشتركة اقتصرت على الصلاحيات المركزية والربط المطلوب.

## الصفحات المنفذة
1. مخزون السيارات مع البحث الجزئي في VIN والفلاتر وPagination من السيرفر.
2. إدارة السيارات وإضافة وتعديل السيارة.
3. استيراد وتصدير Excel مع المعاينة وتقرير الأخطاء.
4. حركة سيارة فردية.
5. حركة جماعية لعدة سيارات إلى مكان واحد مع بيانات وتشيك مستقل لكل سيارة.
6. طلبات النقل والتصوير، الصادر والوارد والمكتمل والملغي.
7. الموافقات المالية والإدارية.
8. جميع السيارات والتقارير المجمعة.
9. سجل الحركات بالفلاتر والتصدير.
10. الأرشيف المشروط.
11. إعدادات العمليات داخل صفحة إعدادات المنصة.

## فلو «مباع تحت التسليم» الإلزامي
- الدخول إلى `مباع تحت التسليم` ينشئ دورة موافقات جديدة ويهيئ المالية والإدارية إلى `false`.
- تعديل بيانات السيارة بدون تغيير الحالة لا يعيد تهيئة الموافقات ولا يمس المستخدمين أو التواريخ أو الملاحظات.
- كارت الداش بورد يعرض فقط سيارات `مباع تحت التسليم`، ويحسب الإجمالي وناقص المالية وناقص الإدارية والمكتملتين.
- الموافقة المالية والإدارية مستقلتان، ولكل منهما ملاحظة ومنفذ وتاريخ وتراجع وسجل تاريخي.
- لا يسمح الـAPI بحالة `مباع تم التسليم` إلا عندما تكون الحالة الحالية `مباع تحت التسليم` والموافقتان مكتملتين معًا.
- اكتمال الموافقتين لا يحول الحالة تلقائيًا؛ يجب تنفيذ حركة أو تغيير حالة رسمي.
- الانتقال الصحيح إلى `مباع تم التسليم` يحافظ على الموافقات وتاريخها.
- الخروج من `مباع تحت التسليم` إلى أي حالة أخرى يمسح الحالة التشغيلية الحالية فقط، مع بقاء Audit Trail وسجل الموافقات السابق.
- جميع مسارات تغيير الحالة، بما فيها التعديل والحركة الفردية والجماعية والاستيراد واستلام طلب النقل، تطبق الشرط داخل Transaction في السيرفر.

## الصلاحيات
تم اعتماد الدور المركزي:
- `system_admin` — مدير النظام.

وتمت إضافة صلاحيات الموافقات المستقلة:
- `operations.approvals.view`
- `operations.approvals.financial`
- `operations.approvals.administrative`
- `operations.approvals.revert`
- `operations.approvals.notes`

مدير النظام يتجاوز قيود الوصول والفروع، لكنه لا يتجاوز سلامة البيانات إلا من خلال Override رسمي بسبب مسجل. مستخدمو الفروع يرون ويتعاملون فقط مع السيارات والمواقع والطلبات التابعة لفروعهم، والتحقق يتم داخل الـAPI. كما تُحسب مؤشرات العمليات في الداش بورد حسب فروع المستخدم، ولا تُعرض بيانات كارت الموافقات بدون صلاحية العرض.

## قاعدة البيانات
ملف الترحيل:
`database/migrations/001_operations_rebuild.sql`

يتضمن:
- نقل مستخدمي الدور القديم `admin` إلى `system_admin` بصورة توافقية.
- مواقع وحالات وسيارات وملاحظات وتشيك مستقل وتاريخ التشيك.
- الموافقات الحالية وسجل الموافقات ودورات الموافقات.
- حفظ سجلات الموافقات القديمة المكررة قبل توحيد السجل الحالي لكل سيارة.
- حركات فردية ودفعات حركة جماعية.
- طلبات النقل والتصوير وسيارات الطلب ومراحل الطلب وسجل الأحداث.
- Locks لمنع تعارض الطلبات، وقيود تمنع VIN المكرر وتكرار المرحلة.
- الأرشيف المنطقي وSnapshot كامل.
- Audit Trail وEvent Outbox.
- Read Model لربط العمليات بالتراكينج بواسطة `vehicle_id` مع fallback منظم للـVIN القديم.
- Indexes على VIN والحالة والمكان والفرع والتواريخ والطلبات والتراكينج.

## خطوات تشغيل Migration
1. أخذ Backup كامل من قاعدة PostgreSQL الحالية.
2. تشغيل الترحيل أولًا على قاعدة Staging:
   `psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f database/migrations/001_operations_rebuild.sql`
3. مراجعة تقرير السيارات وطلبات التراكينج القديمة غير المطابقة قبل الإنتاج.
4. تشغيل اختبارات قبول الفروع والموافقات والحركات والطلبات على Staging.
5. تشغيل الملف نفسه على الإنتاج بعد اعتماد النتائج.

الـAPI يستدعي أيضًا `ensureOperationsSchema()` لضمان وجود المخطط، لكن التشغيل الصريح للـMigration في نافذة نشر مراقبة هو المسار الموصى به.

## التحديث المباشر
أي Mutation ناجحة داخل العمليات تطلق حدث `operations:data-changed`. تستمع إليه:
- الداش بورد الموحدة.
- صفحة الموافقات.
- مخزون السيارات.
- نافذة تفاصيل السيارة.

وبذلك تتحدث العدادات والقوائم والتفاصيل بدون Refresh يدوي داخل الجلسة الحالية.

## حدود الاختبار في بيئة التسليم
- لا يوجد `DATABASE_URL`، لذلك لم تُنفذ اختبارات تكامل حية على PostgreSQL أو ترحيل فعلي لبيانات الإنتاج.
- حزم المشروع غير مثبّتة في البيئة، والاتصال بـ npm registry محجوب؛ لذلك تعذر إكمال `tsc -b` وVite Production Build.
- أخطاء `tsc -b` المسجلة ناتجة عن غياب حزم React وVite وNode type definitions، وليست نتيجة فشل فحوص المصدر القابلة للتشغيل.

بعد فك الملف في بيئة متصلة، يجب تشغيل:
1. `corepack enable`
2. `pnpm install --frozen-lockfile`
3. `pnpm run typecheck`
4. `pnpm run build`
5. اختبارات قبول على قاعدة Staging قبل تحويل المستخدمين.
