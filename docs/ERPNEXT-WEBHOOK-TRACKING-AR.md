# الربط الموحد لطلب البيع من NEXT ERP

هذه النسخة تستخدم **Webhook واحدًا فقط** عند اعتماد طلب البيع، وهو نفس مسار التراكينج الحالي. لا يوجد Webhook إضافي للـCRM أو العمليات، ولا يوجد ربط آلي بحالة `Completed`.

## الفلو النهائي

```text
NEXT ERP — Sales Order / on_submit
              ↓
/api/integrations/erpnext/sales-order
              ↓
التراكينج: إنشاء أو تحديث الطلب والسيارات
              ↓
إذا كانت الحالة To Deliver and Bill وربط المستخدم صحيحًا:
              ├── CRM: العثور على العميل أو إنشاؤه ثم تم البيع
              └── العمليات: العثور على السيارة بالـVIN ثم مباع تحت التسليم
```

## 1. إعداد المستخدمين في المنصة

من:

```text
الإعدادات → المستخدمون والصلاحيات → إضافة/تعديل مستخدم
```

املأ الحقلين الجديدين:

```text
ID المستخدم في NEXT ERP (الإيميل)
اسم الفرع في NEXT ERP
```

- الإيميل هو الـID الثابت الذي يصل من طلب البيع في NEXT ERP.
- اسم الفرع يُكتب كما يظهر حرفيًا في NEXT ERP.
- يجب إدخال الحقلين معًا أو تركهما معًا فارغين.
- الإيميل لا يجوز ربطه بأكثر من مستخدم في المنصة.
- القسم والفرع الأساسيان للمستخدم داخل المنصة يظلان مصدر قسم وفرع CRM.
- لا يتم التخمين من اسم المندوب أو اسم الفرع.

## 2. تحديد العميل الحقيقي

```text
لو اسم المستخدم موجود في طلب البيع:
    العميل الحقيقي = اسم المستخدم
    الجوال = رقم الجوال المرافق له

لو اسم المستخدم فارغ:
    العميل الحقيقي = Customer
    الجوال = رقم الجوال القادم مع الطلب
```

المطابقة في CRM تتم برقم الجوال بعد توحيد صيغته، وليس بالاسم.

- تطابق واحد: تحديث العميل إلى `تم البيع` وربطه بالمندوب والفرع.
- لا يوجد تطابق: إنشاء عميل جديد بحالة `تم البيع`.
- رقم مفقود أو غير صالح: لا يتم إنشاء أو تعديل عميل.
- أكثر من تطابق لنفس الرقم: لا يتم اختيار عميل تلقائيًا ويُعاد تحذير للمراجعة.

## 3. ربط العمليات

لكل سيارة يتم استخدام:

```text
VIN القادم من NEXT ERP = رقم الهيكل في مخزون العمليات
```

عند وجود تطابق واحد وحالة الطلب `To Deliver and Bill`:

```text
حالة السيارة → مباع تحت التسليم
```

قواعد الحماية:

- مكان السيارة الحالي لا يتغير.
- فرع البيع يُحفظ منفصلًا عن مكان السيارة.
- إذا كانت السيارة `مباع تم التسليم` لا يتم إرجاعها إلى حالة سابقة.
- إذا كان VIN غير موجود أو مكررًا في المخزون، لا يتم تعديل سيارة عشوائيًا.
- عند فتح تفاصيل السيارة يظهر تبويب `طلب البيع` بالعميل والمندوب وفرع البيع والمبالغ وروابط CRM والتراكينج.

## 4. التراكينج ومنع التكرار

التراكينج يظل مسار الإدخال الأساسي الحالي. كل سيارة تُعالج من نفس دالة الإدخال الموجودة بالفعل.

```text
الطلب: OrderNo
السيارة داخل الطلب: OrderNo + VIN
```

إعادة إرسال نفس الـWebhook تحدث السجلات الموجودة ولا تنشئ طلبًا أو سيارة مكررة.

## 5. إعداد Webhook في NEXT ERP

```text
Name: MZJ Sales Order Unified Link
DocType: Sales Order
Doc Event: on_submit
Enabled: Yes
Condition: فارغة
Request Method: POST
Request Structure: JSON
Is Dynamic URL: No
Request URL: https://mzj-platform.vercel.app/api/integrations/erpnext/sales-order
Request Timeout: 15
```

Header:

```text
Key: X-MZJ-ERPNext-Key
Value: نفس قيمة ERPNEXT_WEBHOOK_KEY في Vercel
```

استخدم محتوى الملف التالي كاملًا في `JSON Request Body`:

```text
integration-assets/MZJ-ERPNext-Sales-Order-Webhook-JSON.txt
```

## 6. إعداد Vercel

```text
ERPNEXT_WEBHOOK_KEY=مفتاح-سري-طويل
```

بعد إضافته اعمل Redeploy.

## 7. معنى الاستجابة

الاستجابة ترجع `linkage` وفيها:

```text
userLinkStatus      نتيجة مطابقة إيميل وفرع NEXT ERP بالمستخدم
crm.status          نتيجة ربط أو إنشاء عميل CRM
operations.status   نتيجة ربط سيارات مخزون العمليات
warnings            أي بيانات ناقصة أو تعارض يحتاج مراجعة
```

التراكينج يظل محفوظًا حتى لو تعذر ربط المستخدم أو العميل أو VIN؛ لكن CRM والعمليات لا يتم تعديلهما إلا بعد مطابقة آمنة.
