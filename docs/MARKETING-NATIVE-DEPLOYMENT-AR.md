# تشغيل ونشر إعادة بناء نظام التسويق Native

## 1) النسخة الاحتياطية

خذ نسخة احتياطية كاملة من قاعدة PostgreSQL قبل تشغيل ملف SQL.

## 2) تشغيل ملف قاعدة البيانات

من جذر المشروع شغّل:

```bash
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f database/marketing_native_rebuild.sql
```

الملف يعيد إنشاء مخطط `marketing` بصورة منظمة، ولا يحذف مخططات CRM أو التراكينج أو العمليات. كما يضيف جداول طلبات التصوير المشتركة داخل مخطط `operations` عند عدم وجودها.

## 3) متغيرات البيئة

استخدم `.env.example` واضبط متغيرات المنصة الحالية، وبالنسبة للتسويق تأكد من ضبط:

```env
DATABASE_URL=
R2_ACCOUNT_ID=
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_BUCKET=
MZJ_RAW_API_URL=
MZJ_RAW_API_TOKEN=
```

متغيرات R2 مطلوبة لرفع ملفات Task Template والملفات النهائية وملفات النتائج. متغيرا RAW مطلوبان لزر إنشاء فولدرات الخام والتسليم.

## 4) تثبيت الحزم والفحص والبناء

المشروع يستخدم `pnpm@9.15.9`:

```bash
corepack enable
pnpm install --frozen-lockfile
pnpm run typecheck
pnpm run build
```

لا تنشر النسخة إذا فشل `typecheck` أو `build`.

## 5) النشر

انشر سورس المنصة كاملًا بنفس آلية نشر المنصة الحالية بعد نجاح SQL والفحوصات والبناء. لا توجد خدمة Firebase أو Publisher محلي خاصة بالتسويق، ولا يوجد Route مستقل لإعدادات التسويق؛ الإعدادات موجودة داخل صفحة الإعدادات الموحدة.

## 6) اختبار ما بعد النشر

بعد النشر اختبر بحساب مدير النظام وحساب كاتب محتوى وحساب يوزر تنفيذي:

- فتح إعدادات التسويق وإدارة الأقسام والكرييتيفات والإجراءات والمنصات والتصنيفات.
- إنشاء حملة بخطواتها الخمس وإنشاء أجندة بخطواتها الثلاث.
- رفع واعتماد وطلب تعديل ورفض Task Template.
- التأكد من أن اعتماد Template يفتح التاسكات التنفيذية المرتبطة بها فقط.
- تسجيل «تم الاستلام» ومراجعة انتقال التاسك والتقويم.
- إكمال إجراءات التكليف حتى 100% ثم النقل إلى قسم النشر.
- إنشاء طلب تصوير من الاستوك والتأكد من ظهوره في التسويق وداخل متابعة الطلبات في العمليات.
