# تنفيذ نظام العمليات Native داخل منصة MZJ

## المصدر المعتمد

تم التنفيذ داخل سورس `MZJ-Platform-v1.13.2-Tracking-FULL` فقط. استخدم سورس العمليات القديم لفهم الفلو والحقول والحالات، ولم يتم نقل صفحات HTML أو Firebase أو JavaScript أو CSS أو Routes أو APIs منه.

## النطاق المنفذ

- وحدة عمليات React + TypeScript داخل القائمة والتوجيه الحاليين للمنصة.
- API موحدة في `server/operations/index.ts` مع Service Layer في `server/_operations-service.ts`.
- PostgreSQL Schema ومهاجرة منظمة في `database/migrations/20260719_operations_native.sql`.
- صلاحيات Server-Side مركزية، مع دعم الدور الأعلى `system_admin`.
- مخزون السيارات، تفاصيل السيارة، الإدارة، الاستيراد والتصدير، والبحث الجزئي في VIN.
- تبويب حركة موحد لسيارة واحدة أو عدة سيارات داخل Transaction ذرية.
- تشيك مستقل لكل سيارة، ويظهر في الحركة فقط عندما يكون موقع السيارة الحالي `الوكالة` (`agency`).
- طلبات النقل والتصوير، مركبات الطلب، المراحل، الصادر والوارد، الحذف قبل بدء التنفيذ، والإلغاء بعد بدء التنفيذ.
- تجاوز إداري صريح لمدير النظام فقط، مع سبب إلزامي وتسجيل Audit Trail.
- الموافقات المالية والإدارية المستقلة ودورة `مباع تحت التسليم` ثم `مباع تم التسليم`.
- منع التسليم النهائي من الـAPI ما لم تكتمل الموافقتان ولم يتم التراجع عن أي منهما.
- سجل الحركات، Batch ID للحركة الجماعية، سجل التشيك، سجل الموافقات، وسجل أحداث الطلبات.
- الأرشيف المنطقي دون حذف السيارة أو تاريخها.
- تكامل Tracking باستخدام `operations_vehicle_id` أولًا، ثم VIN للنماذج القديمة مع الحفاظ على الأصفار.
- Event Outbox لتجهيز نظام الإشعارات المركزي مستقبلًا.
- توحيد مصادر بيانات الداش بورد وصفحات العمليات.
- جميع النوافذ الجديدة تستخدم آلية الإغلاق بـ Escape ومكونات المنصة الحالية.

## قواعد سلامة البيانات

- VIN مخزن كنص مع Unique Constraint وIndexes للبحث.
- منع تكرار VIN ومنع تكرار السيارة داخل الطلب أو الحركة.
- قفل صفوف السيارات والطلبات في الإجراءات الحساسة.
- تحديث السيارة وإنشاء الحركة وسجل التشيك والموافقات داخل Transactions.
- الحركة الجماعية Atomic: نجاح كامل أو Rollback كامل.
- لا يسمح بطلب واحد يخلط سيارات من أكثر من موقع مصدر أو أكثر من فرع مصدر.
- لا يسمح بإنشاء طلب متعارض لسيارة لديها طلب نشط.
- لا يسمح بحذف طلب بدأ تنفيذه؛ يستخدم الإلغاء مع السبب.
- لا يتم تعديل أو حذف Audit Trail من الواجهة.
- الاستيراد لا يمسح القيمة القديمة عندما تكون خلية التحديث فارغة.

## الصلاحيات الأساسية

- `system_admin`: جميع الأنظمة والفروع والمواقع والعمليات، مع بقاء قواعد سلامة البيانات.
- `accounting_manager`: الموافقة المالية والتراجع عنها.
- `operations_manager`: الموافقة الإدارية والتراجع عنها.
- بقية الإجراءات تعتمد على Permissions من Session وتفحص مرة أخرى داخل الـAPI.

الصلاحيات المستخدمة تشمل:

- `operations.view`
- `operations.vehicles.view/create/update/import/export`
- `operations.movements.create`
- `operations.requests.view/create/progress/cancel`
- `operations.approvals.view/financial/administrative`
- `operations.archive.view/create`
- `operations.tracking.view/open`
- `operations.audit.view`
- `operations.settings.manage`

## الصفحات والمسارات

- `/operations` مخزون السيارات.
- `/operations/vehicles` إدارة السيارات.
- `/operations/movement` الحركة الموحدة.
- `/operations/requests` طلبات النقل والتصوير.
- `/operations/approvals` الموافقات.
- `/operations/movements` سجل الحركات.
- إعدادات العمليات داخل صفحة إعدادات المنصة الحالية.

لا توجد داش بورد عمليات مستقلة، ولا Login منفصل، ولا Users منفصلون، ولا Tracking أو Media قديمة داخل العمليات.
