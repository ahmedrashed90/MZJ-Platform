# تشغيل ونشر Operations Native V2

## 1. قبل النشر

1. أنشئ Snapshot/Backup كاملًا لقاعدة PostgreSQL.
2. سجّل أعداد السيارات وVINs والحركات والطلبات والموافقات وطلبات التراكينج.
3. أنشئ قاعدة Staging من نفس الـSnapshot.
4. تأكد أن إصدار التراكينج الحالي في الـBaseline مطبق، لأن Migration العمليات توسع جداول التراكينج الموجودة.

## 2. تثبيت وبناء المنصة

```bash
corepack enable
pnpm install --frozen-lockfile
pnpm run typecheck
pnpm run build
```

يجب ألا يتم النشر إذا فشل أي أمر.

## 3. تطبيق Migration على Staging

```bash
psql "$STAGING_DATABASE_URL" -f database/migrations/20260720_operations_native_v2.sql
```

تطبيق الـSchema يتم أيضًا بصورة Idempotent عند تهيئة منصة جديدة أو أول استدعاء لمسارات العمليات، لكن تشغيل Migration المراجعة يدويًا على Staging هو المسار المفضل قبل Main.

## 4. اختبارات القبول على Staging

### الاتصال والداش بورد

- افصل أو اكسر Query لقسم اختباريًا وتأكد أن بقية الداش بورد تظهر.
- تأكد أن رسالة PostgreSQL غير مربوط لا تظهر إلا عند فشل الاتصال الحقيقي.
- افتح كل مؤشر فرع وقارن العدد مع عدد النتائج والتصدير.

### Excel والاستيراد

- نزّل القالب وتأكد أنه ملف `.xlsx` حقيقي ويفتح في Excel.
- ارفع ملف XLSX يحتوي VIN يبدأ بأصفار وتأكد من بقائها.
- اختبر أوضاع: إضافة الجديد فقط، تحديث الموجود فقط، والاستبدال داخل نطاق الفرع.
- أضف صفًا خاطئًا وسط صفوف صحيحة وتأكد من ظهور تقرير الصف دون إفساد بقية الدفعة.

### الحركة

- حركة سيارة واحدة.
- حركة عدة سيارات.
- فشل سيارة واحدة يؤدي إلى Rollback كامل.
- تحقق من ظهور الحركات في سجل الحركات مباشرة.

### طلب النقل

- إنشاء طلب لسيارة واحدة ثم لعدة سيارات من نفس المكان المصدر.
- منع Duplicate Submit وطلب متعارض.
- تنفيذ المراحل بالترتيب.
- تحقق أن استلام السيارة يحدث الموقع والحركة والمرحلة معًا.

### الموافقات

- إدخال سيارة في مباع تحت التسليم.
- موافقة مالية فقط ثم رفض مباع تم التسليم.
- موافقة إدارية فقط ثم الرفض.
- اكتمال الموافقتين ثم تنفيذ حركة رسمية لمباع تم التسليم.
- تعديل بيانات أخرى وعدم تصفير الموافقات.

### مسح السيارة

- سيارة جديدة بلا تاريخ: تنجح.
- سيارة لها حركة/موافقة/طلب/Tracking/أرشيف/Outbox: تُرفض بـ`VEHICLE_HAS_HISTORY`.
- تحقق من بقاء `operations.vehicle_deletion_audit` بعد الحذف.

### مسح طلب التراكينج

- نافذة التأكيد تظهر فوق التفاصيل.
- رقم طلب غير مطابق أو سبب فارغ يمنع التنفيذ.
- حذف طلب سيارة واحدة ومتعدد السيارات.
- لا تُحذف سيارات مخزون العمليات.
- السيارة بلا طلب آخر تظهر `لا يوجد طلب`.
- نفس source identity القديمة لا تعود.
- صف Google Sheet يبقى `PlatformSynced = YES` ولا يُحذف.
- طلب جديد بنفس `sales_order_no` من source identity جديدة يُقبل.

## 5. تطبيق Main

بعد اعتماد نتائج Staging فقط:

```bash
psql "$DATABASE_URL" -f database/migrations/20260720_operations_native_v2.sql
pnpm run build
```

ثم انشر المنصة واختبر Smoke Test بحساب system_admin وحساب مقيد بفرع.

## 6. التراجع

لا تشغل Drop للجداول أو الأعمدة. عند وجود مشكلة حرجة:

1. أوقف الكتابة.
2. أعد نشر الإصدار السابق.
3. استعد Snapshot ما قبل Migration إلى قاعدة منفصلة.
4. قارن البيانات الجديدة قبل أي Switch لـ`DATABASE_URL`.
5. استخدم ملف rollback فقط لتعطيل الصلاحيات الجديدة مؤقتًا؛ لا يعتبر استعادة للبيانات.
