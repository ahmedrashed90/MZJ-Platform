# تقرير تنفيذ الربط الموحد لطلب البيع — v1.16.4

## مبدأ التنفيذ

تم تطوير الربط داخل آخر سورس كأساس واحد، وليس بإضافة Webhooks أو مسارات متوازية فوق بعضها. نقطة الدخول الوحيدة هي:

```text
POST /api/integrations/erpnext/sales-order
```

نفس الطلب يمر بالترتيب على التراكينج، ثم خدمة الربط الموحدة للـCRM والعمليات.

## المنفذ

- إضافة حقل `ID المستخدم في NEXT ERP (الإيميل)` وحقل `اسم الفرع في NEXT ERP` داخل إضافة وتعديل المستخدم، مع إلزام إدخالهما معًا أو تركهما معًا فارغين.
- مطابقة المندوب باستخدام إيميل NEXT ERP، والتحقق من تطابق فرع NEXT ERP المحفوظ.
- استخدام قسم وفرع المستخدم الأساسيين في المنصة عند تحديث CRM.
- تحديد العميل الحقيقي من `اسم المستخدم` إن كان موجودًا، وإلا من `Customer`.
- مطابقة CRM برقم الجوال فقط.
- تحديث العميل الموجود إلى `تم البيع` أو إنشاء عميل جديد بالحالة نفسها.
- منع أي تعديل في CRM عند غياب الجوال أو وجود تطابقات متعددة.
- الإبقاء على دالة إدخال التراكينج الحالية كمسار الكتابة الوحيد.
- ربط مخزون العمليات بالـVIN فقط.
- تحويل السيارة إلى `مباع تحت التسليم` عند `To Deliver and Bill`.
- عدم تغيير الموقع الحالي للسيارة أو إنشاء طلب نقل.
- الحفاظ على `مباع تم التسليم` وعدم إرجاع السيارة إلى حالة سابقة.
- إضافة تبويب `طلب البيع` داخل تفاصيل السيارة.
- إضافة جداول تكامل مرجعية تربط طلب البيع بالعميل والتراكينج وسيارة العمليات.
- إضافة Migration مستقل متوافق مع قواعد البيانات الحالية، مع ضمان إنشاء الجداول Runtime أيضًا.
- لا توجد معالجة آلية لحالة `Completed` ولا Webhook ثانٍ.

## الهيكل البرمجي

```text
server/_erpnext-sales-order-normalizer.ts
    مسؤول فقط عن تحويل بيانات NEXT ERP إلى عقد داخلي موحد

server/integrations/erpnext-sales-order.ts
    Endpoint صغير للمصادقة وتشغيل التراكينج ثم الربط الموحد

server/_erpnext-sales-order-sync.ts
    Orchestrator واحد لربط المستخدم وCRM ومخزون العمليات

server/_erpnext-integration-schema.ts
    مخطط قاعدة البيانات الخاص بالتكامل
```

هذا الفصل يمنع تكرار منطق الإدخال ويجعل التحويل والربط والمصادقة مسؤوليات منفصلة وواضحة.

## الاختبارات

- فحوصات الموديولات الحالية للـCRM والتراكينج والعمليات.
- فحص ثابت جديد للربط الموحد v1.16.4.
- فحص Syntax للملفات TypeScript/TSX المعدلة.
- اختبار Normalizer على معاينة طلب حقيقي يحتوي على 15 سيارة.
- اختبار حالتي العميل: `Customer` مباشرة، و`اسم المستخدم + رقم الجوال`.
- اختبار استخراج الحقول الفعلية: `serial_no`, `type`, `class`, `model`, `custom_internal_color`, `custom_external_color`, `agent`.

## ملاحظة البناء

لم يتم تغيير Dependencies أو Lockfile. تشغيل التثبيت الكامل للحزم يحتاج اتصالًا بسجل npm؛ لذلك تعتمد المراجعة المحلية على الفحوصات الثابتة وفحص TypeScript المتاح واختبارات التحويل المباشرة.
