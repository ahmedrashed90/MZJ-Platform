# تقرير تنفيذ الربط الموحد لطلب البيع — v1.16.6

## مبدأ التنفيذ

تم اعتماد مسار واحد فقط لطلبات البيع بدون Webhook إضافي أو منطق موازٍ:

```text
POST /api/integrations/erpnext/sales-order
```

يمر الطلب على التراكينج ثم ربط CRM ومخزون العمليات من نفس خدمة المعالجة.

## ربط المستخدم

- حلقة الربط الوحيدة هي `ID المستخدم في NEXT ERP (الإيميل)` داخل المستخدم المحفوظ في المنصة.
- لا يوجد حقل أو شرط لمطابقة اسم فرع NEXT ERP مع فرع المنصة.
- `Branch` القادم من طلب البيع يُحفظ كبيان لفرع البيع ويظهر في تفاصيل السيارة فقط.
- قسم المستخدم وفرعه الأساسيان داخل المنصة يظلان المصدر التنظيمي المستخدم في CRM.

## CRM

- العميل الحقيقي هو `اسم المستخدم` عند وجوده، وإلا `Customer`.
- المطابقة تتم برقم الجوال بعد توحيده.
- لو الرقم موجود مرة واحدة: يتم تحديث نفس العميل إلى `تم البيع`.
- لو الرقم غير مسجل: يتم إنشاء عميل جديد تلقائيًا بحالة `تم البيع` وربطه بالمندوب.
- لو الرقم فارغ/غير صالح أو مكرر على أكثر من عميل: لا يتم تعديل CRM بالاسم فقط، ويستمر التراكينج والعمليات مع تحذير واضح.

## التراكينج والعمليات

- دالة إدخال التراكينج الحالية هي مسار الكتابة الوحيد.
- ربط سيارة العمليات يتم بالـVIN فقط.
- `To Deliver and Bill` يحول السيارة إلى `مباع تحت التسليم`.
- موقع السيارة الحالي لا يتغير، حتى لو فرع البيع مختلف.
- لا توجد معالجة آلية لـ`Completed` ولا Webhook ثانٍ.

تم تنظيف نموذج المستخدم وAPI المستخدمين ومخطط قواعد البيانات من ربط فرع NEXT ERP بدل إضافة استثناءات فوق المنطق السابق.
