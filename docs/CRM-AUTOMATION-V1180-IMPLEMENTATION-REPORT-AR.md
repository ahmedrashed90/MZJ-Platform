# تقرير تنفيذ نظام أوتوميشن المحادثات — MZJ CRM v1.18.0

## 1. نطاق التنفيذ

تم التنفيذ على سورس منصة MZJ المرفوع باعتباره المصدر الأساسي، مع إعادة بناء فلو أوتوميشن المحادثات داخل المنصة وربطه بمحرك دخول وتوزيع العملاء الحالي، وتحويل Facebook Worker إلى طبقة نقل تقنية فقط.

لم يتم تعديل منطق أنظمة العمليات أو التراكينج أو ERPNext أو التقارير خارج نقاط التكامل الضرورية، ولم يتم إنشاء محرك توزيع جديد داخل الأوتوميشن أو الـWorker.

## 2. المعمارية النهائية

### المنصة هي المصدر الوحيد للحقيقة

المنصة تملك الآن:

- تشغيل وإيقاف الأوتوميشن.
- سياسة التشغيل: كل رسالة، مرة كل 24 ساعة، أو مدة مخصصة.
- رسائل البداية وترتيبها.
- اختيارات الخدمات والـPayload والردود المقبولة.
- خطوات الأسئلة والتحقق وحفظ الإجابات.
- جلسات الأوتوميشن وحالتها الحالية.
- إنشاء أو تحديث العميل وطلب الخدمة.
- استدعاء محرك التوزيع المركزي الحالي.
- تسجيل الإجراء النهائي ورسالة النهاية.
- منع التكرار على مستوى الحدث، الرسالة، الجلسة، الإرسال، والنتيجة النهائية.

### مسؤولية Facebook Worker

Facebook Worker أصبح مسؤولًا فقط عن:

- تحقق Meta Webhook والتوقيع.
- استقبال النصوص وQuick Replies وPostbacks والمرفقات.
- استخدام Facebook PSID كهوية المحادثة الأساسية.
- تمرير الحدث كاملًا إلى منصة MZJ.
- إرسال النصوص والأزرار والوسائط عبر Facebook Graph API.
- ManyChat كـfallback اختياري لإرسال النصوص فقط.
- إعادة نتيجة إرسال حقيقية: sent أو failed مع Provider Message ID وHTTP Status.

مسار `/automation` القديم محفوظ للتوافق، لكنه لا يمرر نسخة ثانية من رد العميل. يتم تأجيله دائمًا إلى Meta Webhook لمنع تشغيل Meta وManyChat كفلوين متوازيين.

## 3. السيناريوهات الافتراضية

### مبيعات الكاش

1. العميل يختار `💰 مبيعات الكاش`.
2. يتم إنشاء أو تحديث العميل وطلب الكاش مرة واحدة.
3. يتم استدعاء محرك التوزيع المركزي لقسم مبيعات الكاش.
4. يتم إرسال:

```text
تم تحويل طلبك إلى قسم مبيعات الكاش ✅
سيتم التواصل معك قريباً
```

### مبيعات التمويل

الفلو يعمل بالترتيب الإجباري:

1. إرسال `برجاء إدخال بيانات التمويل 👇\nالاسم`.
2. حفظ الاسم وتثبيته بحيث لا يستبدله اسم Facebook لاحقًا.
3. إرسال `السيارة` وحفظ الإجابة.
4. إرسال `رقم الجوال`.
5. التحقق من رقم سعودي صحيح وتطبيعه.
6. تطبيق منع التكرار ودمج جهة الاتصال عند وجود نفس الجوال.
7. إنشاء أو تحديث عميل التمويل وطلبه.
8. توزيع مندوب التمويل.
9. توزيع مندوب الكول سنتر.
10. إرسال:

```text
سيتم التواصل معك في أقرب وقت
نسعد بخدمتكم دائمًا 🌹
```

لا يتم إنشاء عميل تمويل جديد أو توزيع الطلب قبل اكتمال رقم الجوال الصحيح.

### خدمة العملاء

1. العميل يختار `🛠 خدمة العملاء`.
2. يتم إنشاء أو تحديث العميل وطلب خدمة العملاء.
3. يتم استدعاء قواعد توزيع خدمة العملاء الحالية فقط.
4. يتم إرسال:

```text
سيتم التواصل معك قريباً من أحد ممثلي قسم خدمة العملاء 👨‍🔧
```

## 4. صفحة إعدادات الأوتوميشن

تمت إضافة تبويب:

`الإعدادات → إعدادات CRM → إعدادات الأوتوميشن`

ويحتوي على:

- الحالة العامة واسم الأوتوميشن.
- سياسة التشغيل والمدة المخصصة.
- المنصات والـWorkers وحالة الربط وHealth Check.
- منشئ رسائل البداية مع الإضافة والتعديل والحذف والترتيب والتفعيل.
- منشئ الاختيارات والردود المقبولة.
- منشئ خطوات الفلو: رسالة فقط، سؤال نصي، رقم جوال، اختيار.
- أقل وأقصى طول، القيم المقبولة، رسالة خطأ التحقق، وعدد المحاولات.
- الإجراء النهائي ورسالة النهاية.
- سجل آخر 100 جلسة وحالة التوزيع.

صلاحية الصفحة والـAPI مطبقة لمستخدمي إدارة CRM فقط، وليس بإخفاء الواجهة وحدها.

## 5. الفصل عن دخول وتوزيع العملاء

تم إيقاف تعديل رسائل واختيارات الدخول من المسار القديم `/api/crm/entry-routing` حتى لا توجد إعدادات متوازية.

ظل محرك التوزيع المركزي مسؤولًا عن:

- قواعد الموظفين المؤهلين.
- ترتيب الدور وRound Robin.
- الأقسام والفروع.
- توزيع مندوب المبيعات.
- توزيع الكول سنتر.
- تسجيل نتيجة التوزيع.

الأوتوميشن لا يختار الموظف بنفسه؛ يستدعي نفس خدمات التوزيع الحالية داخل نفس معاملة قاعدة البيانات.

يوجد ملف Rollback مستقل وملف Post-check للقراءة فقط بعد النشر.

## 7. منع التكرار والتزامن

تم تطبيق:

- Advisory transaction lock لكل محادثة.
- معرف حدث ثابت لكل رسالة واردة.
- Unique provider message ID داخل المنصة.
- جلسة نشطة واحدة لكل عميل ومحادثة.
- Idempotency Key لكل رسالة أو سؤال أو رسالة نهاية.
- سجل إجراء نهائي واحد لكل جلسة.
- إعادة محاولة السؤال أو الرسالة النهائية بنفس المعرف بدون إنشاء فلو أو توزيع جديد.
- منع الاستهلاك المزدوج لنفس رسالة العميل عند فشل إرسال السؤال التالي.
- منع تطابق الرقم `1` داخل رقم جوال على أنه اختيار كاش؛ الرد الرقمي يجب أن يطابق تمامًا.
- منع إرسال Graph مرتين في Facebook Worker لنفس معرف الإرسال، مع KV موصى به للحفظ الدائم.

## 8. معالجة الأخطاء

- فشل إرسال سؤال لا ينقل الجلسة للخطوة التالية باعتبارها ناجحة.
- فشل حفظ الإجابة يمنع السؤال التالي.
- فشل إنشاء العميل يمنع التوزيع ورسالة النهاية.
- فشل رسالة النهاية يُسجل ويمكن إعادة محاولته بنفس الجلسة والإجراء.
- عدم وجود موظف متاح لا ينشئ توزيعًا وهميًا؛ يتم تسجيل `waiting_assignment` في الإجراء النهائي وداخل Metadata طلب الخدمة.
- جميع الرسائل الواردة تظل محفوظة في المحادثة حتى إذا توقف الأوتوميشن أو فشل أثر جانبي.

## 9. المنطق القديم الذي تمت إزالته أو تعطيله

- إزالة فلو اختيار الخدمة القديم من `server/_crm-automation.ts`.
- منع `server/_integration-processor.ts` من تنفيذ تصنيف خدمة موثوق موازٍ للقنوات التي يملكها محرك المحادثات الجديد.
- منع `/api/crm/entry-routing` من حفظ مصدر ثانٍ للرسائل والاختيارات.
- حذف منطق تحديد كاش/تمويل/خدمة العملاء وإنشاء العميل والتوزيع من Facebook Worker الجديد.
- تعطيل ManyChat كمسار دخول موازٍ مع الإبقاء على Route للتوافق.

لم يتم حذف ملفات أنظمة أخرى، ولم يتم إنشاء دوال Fix مؤقتة أو جداول ترقيعية.

## 10. اختبارات التنفيذ

نجحت الاختبارات التالية في بيئة الفحص:

- فحص نحوي لملفات TypeScript وTSX المعدلة.
- فحص تطابق Migration المضمنة مع ملف Migration المنشور.
- فحص الجداول والفلو والرسائل والترتيب ومنع التكرار.
- اختبار Offline لطلبات واستجابات Facebook Worker.
- اختبار Quick Replies والـPayload.
- اختبار Provider Message ID وsent/failed.
- اختبار عدم إرسال Graph مرتين لنفس Idempotency Key.
- اختبار أن ManyChat لا يبدأ فلوًا موازيًا حتى عند وجود PSID.
- جميع اختبارات الرجوع الموجودة في CRM والعمليات والتراكينج وERPNext.

### ما لم يتم اختباره حيًا

لم يتم تشغيل اختبار End-to-End على قاعدة PostgreSQL الفعلية أو حساب Meta الحقيقي، لأن بيئة الفحص لا تحتوي `DATABASE_URL` أو أسرار Meta/Cloudflare أو وصولًا للشبكة الخارجية. لذلك يلزم بعد النشر تنفيذ اختبارات القبول الحية الثلاثة من حساب Facebook فعلي ومراجعة الجلسات والتوزيعات في المنصة.
