# إعادة بناء التسويق Native النظيفة — v1.20.0

## القرار المعماري

تم تنفيذ هذه النسخة من سورس المنصة الأساسي v1.18.0، وليس من سلسلة v1.19.x التي احتوت محاولات توافق متراكمة مع جداول التسويق القديمة.

نظام التسويق الجديد يستخدم PostgreSQL Schema مستقلة باسم:

```text
marketing_native
```

لا يقرأ التنفيذ الجديد جداول الدومين القديمة داخل `marketing`، ولا يغيّرها، ولا ينفذ عليها Seeds أو ALTER TABLE. الهدف هو منع أي قيد قديم أو نوع عمود قديم أو Foreign Key قديم من التأثير في التشغيل الجديد.

## سبب خطأ platform_post_types السابق

قاعدة الإنتاج احتوت جدولًا قديمًا باسم `marketing.platform_post_types` وعلاقة قديمة لا تتوافق مع طريقة إنشاء المنصات في التنفيذ الجديد. كان إدخال أنواع النشر يصطدم بقيد:

```text
platform_post_types_platform_id_fkey
```

في v1.20.0 لم يعد هذا الجدول جزءًا من مسار التشغيل. يتم إنشاء الأب والابن داخل نفس الـSchema ونفس المعاملة:

```text
marketing_native.platforms
marketing_native.platform_post_types
```

وتُستخرج قيمة `platform_id` في Seed أنواع النشر مباشرة من `marketing_native.platforms` باستخدام `platforms.code`. لذلك لا يمكن أن يحصل Seed الطفل على ID من الـSchema القديمة.

## ضمانات التهيئة

- كل DDL وSeeds داخل Transaction واحدة.
- PostgreSQL Advisory Lock يمنع تهيئتين متزامنتين في Vercel.
- إنشاء جدول المنصات يسبق إنشاء وجدولة أنواع النشر.
- كل Foreign Key داخلي يشير إلى جدول تم تعريفه قبله.
- فحص قبل Commit يتأكد من:
  - وجود 29 جدولًا أساسيًا.
  - رقم العقد 1200.
  - عدم وجود صف orphan في `platform_post_types`.
  - توجيه FK الخاص بأنواع النشر إلى `marketing_native.platforms` تحديدًا.
  - عدم وجود أي FK من `marketing_native` إلى الـSchema القديمة `marketing`.
- أي فشل يؤدي إلى Rollback كامل؛ لا تظل قاعدة البيانات نصف مهيأة.
- Seeds الإعدادات غير تدميرية (`ON CONFLICT DO NOTHING`) ولا تعيد كتابة تعديلات الإدارة عند كل Cold Start.

## التوافق مع بقية المنصة

تم الحفاظ على CRM والتراكينج والعمليات والـERPNext والوركرز من سورس v1.18.0، مع التعديلات الضرورية فقط لربط طلبات التصوير المشتركة وإظهار نظام التسويق في الـRouter والإعدادات والداش بورد.

طلبات التصوير تستخدم سجل العمليات نفسه، وتتم تهيئة Schema العمليات قبل Schema التسويق داخل API التسويق.

## طريقة النشر

1. ارفع مجلد السورس الكامل من هذا الإصدار.
2. نفّذ Redeploy بدون Build Cache.
3. لا تشغّل SQL يدويًا.
4. لا تنقل ملفات من v1.19.x إلى هذه النسخة.
5. لا تحذف الـSchema القديمة يدويًا؛ التنفيذ الجديد يتجاهلها ولا يعتمد عليها.

## حدود التحقق داخل بيئة التسليم

تم تشغيل TypeScript والفحوص الثابتة ومحاكاة التهيئة بالكامل. لم تتوفر إمكانية اتصال بقاعدة PostgreSQL الإنتاجية أو تشغيل PostgreSQL محلي داخل بيئة التنفيذ. كذلك تعذر تنزيل حزم npm لإعادة Vite bundle فعلي بسبب فشل DNS إلى `registry.npmjs.org`. لذلك لا يوجد ادعاء بأن قاعدة الإنتاج نفسها تم اختبارها من داخل هذه البيئة.

التصميم المعزول يقلل الاعتماد على الحالة التاريخية لقاعدة الإنتاج: يحتاج فقط إلى جداول `core` و`operations` الأصلية الموجودة في المنصة، ويُنشئ كل دومين التسويق الجديد داخل `marketing_native` من الصفر بصورة ذرية.
