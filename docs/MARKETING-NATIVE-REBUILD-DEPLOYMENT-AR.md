# تعليمات SQL والنشر — إعادة بناء التسويق Native

## 1. النسخ الاحتياطي

قبل التنفيذ:

1. خذ نسخة احتياطية من قاعدة PostgreSQL.
2. احتفظ بنسخة من سورس المنصة المنشور حاليًا.
3. نفّذ الخطوات أولًا على بيئة Staging بنفس Schema الإنتاج.

## 2. متغيرات البيئة

احتفظ بكل متغيرات المنصة الحالية، وتأكد خصوصًا من متغيرات قاعدة البيانات والجلسات والتخزين. التكامل الاختياري لفولدرات الخام يستخدم:

```env
MZJ_RAW_API_URL=
MZJ_RAW_API_TOKEN=
```

إذا لم يتم ضبطهما، تظل الحملة محفوظة وتظهر رسالة أن خدمة إنشاء الفولدرات غير مفعلة؛ لا يتم التراجع عن إنشاء الحملة.

## 3. تشغيل Migration

الملف النهائي المنظم:

```text
database/migrations/20260723_marketing_native_rebuild.sql
```

تشغيله باستخدام `psql`:

```bash
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f database/migrations/20260723_marketing_native_rebuild.sql
```

الملف قابل لإعادة التشغيل بأمان ويقوم بالآتي:

- إنشاء مخطط وجداول وفهارس وصلاحيات التسويق.
- إضافة Seed أولي قابل للتعديل من الإعدادات للأقسام والكرييتيفات والمنصات والحالات والتصنيفات.
- إنشاء جدول سجل متابعة طلبات التصوير المشترك داخل العمليات.
- لا يحذف بيانات CRM أو التراكينج أو العمليات.
- لا يحتوي على DROP عشوائي.

## 4. تثبيت الاعتماديات

استخدم Node.js 22 ومدير الحزم المثبت في المشروع:

```bash
corepack enable
pnpm install --frozen-lockfile
```

## 5. الاختبارات الإلزامية قبل النشر

```bash
pnpm run test:marketing
pnpm run typecheck
pnpm run build
```

لا تنشر النسخة إذا فشل أي أمر.

## 6. نشر المنصة

بعد نجاح Build:

```bash
pnpm run build
vercel --prod
```

أو استخدم Pipeline النشر المعتمد للمنصة مع نفس متغيرات البيئة.

## 7. ضبط الصلاحيات

صلاحيات التسويق تُضاف إلى `core.permissions`، وتُمنح تلقائيًا لأدوار `admin` و`system_admin`. لباقي الأدوار استخدم صفحة المستخدمين والصلاحيات في المنصة، وامنح فقط المطلوب، مثل:

- `marketing.view`
- `marketing.project.create`
- `marketing.project.edit`
- `marketing.task.receive`
- `marketing.task.execute`
- `marketing.template.upload`
- `marketing.template.review`
- `marketing.publish.manage`
- `marketing.package.manage`
- `marketing.stock.view`
- `marketing.photo_request.create`
- `marketing.photo_request.manage`
- `marketing.reports.view`
- `marketing.attendance.use`
- `marketing.attendance.manage`
- `marketing.connections.manage`
- `marketing.settings.manage`

## 8. إعداد بيانات التسويق

من الإعدادات الموحدة افتح «إعدادات التسويق» وراجع:

1. الأقسام وقسم المحتوى.
2. ربط المستخدمين بالأقسام.
3. الكرييتيفات والقسم الأساسي والكود المختصر.
4. إجراءات التكليف والنسب والجمهور.
5. أنواع الحملات وتسلسل الأكواد.
6. المنصات وأنواع النشر والأبعاد.
7. حالات طلبات التصوير.
8. تصنيفات الباقات.
9. مواعيد الدوام وفترة السماح.

لا تبدأ إنشاء حملات فعلية قبل اكتمال ربط اليوزرات بالأقسام.

## 9. Smoke Test بعد النشر

نفّذ بالترتيب:

1. إنشاء حملة تجريبية بها Instance واحدة وكاتبا محتوى ويوزران تنفيذيان.
2. التأكد من إنشاء Template مستقلة لكل كاتب، وتاسك مستقل لكل علاقة.
3. اعتماد Template واحدة والتأكد أن التاسكات غير المرتبطة تظل مغلقة.
4. الضغط على «تم الاستلام» والتأكد من ظهور التاريخ الفعلي في التقويم.
5. إكمال إجراءات التكليف والتأكد من حساب التقدم.
6. رفع ملف نهائي والتأكد من ظهوره تحت التاسك والـInstance الصحيحين.
7. الوصول إلى 100% ثم نقل الحملة إلى قسم النشر.
8. إنشاء أجندة في يوم واحد ثم في عدة أيام والتحقق من حدود التاريخ.
9. إنشاء طلب تصوير من الاستوك والتأكد من ظهوره في التسويق والعمليات.
10. تحديث حالة طلب التصوير من العمليات والتأكد من ظهورها فورًا في التسويق وسجل المتابعة.
11. اختبار الباقات والتصنيفات والعدّادات.
12. اختبار الحضور والانصراف بصلاحيات مستخدم وأدمن.

## 10. التراجع

لا يوجد DROP تلقائي في Migration. عند الحاجة للتراجع:

- أعد نشر نسخة السورس السابقة.
- استعد نسخة قاعدة البيانات التي أخذتها قبل Migration إذا كان مطلوبًا إزالة بيانات التسويق الجديدة.
- لا تحذف جداول `operations.photography_requests` لأنها سجلات مشتركة وقد تحتوي على بيانات تشغيلية فعلية.
