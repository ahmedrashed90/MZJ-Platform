# تقرير تحديث CRM داخل منصة MZJ الموحدة — Reference v39

## 1) نطاق العمل والمصادر

تم اعتماد سورس المنصة الموحدة كأساس وحيد للتعديل، واستخدام سورس CRM المستقل كمرجع وظيفي للمقارنة فقط. لم يتم نسخ صفحات أو ملفات JavaScript/Firebase من CRM المستقل، ولم يتم إنشاء تنفيذ موازٍ أو ملفات بديلة بأسماء `new` أو `fixed` أو `v2`.

الإصدار الناتج: **MZJ Platform 1.15.0**.

## 2) نتيجة Inventory والمقارنة الوظيفية

### أجزاء كانت موجودة بالفعل في المنصة وتم الحفاظ عليها

- React + TypeScript + PostgreSQL وAPI المنصة.
- تسجيل الدخول الموحد ومستخدمي وصلاحيات المنصة.
- الفروع والأقسام والتوزيع التلقائي وقواعد Round Robin.
- صندوق الوارد الموحد والرسائل غير المقروءة والوسائط وملفات PDF.
- منع تكرار أرقام الجوال وفلو موافقة العميل المكرر.
- سجل العميل، سجل الملكية، سجل تغيير الحالات، وAudit Log.
- داش بورد CRM الموحد مع الكاش والتمويل وخدمة العملاء.
- سجل عملاء التمويل وتكامل بيانات CRM مع الداش بورد الموحدة.
- إدارة القوالب، الحالات، الفروع، قواعد التوزيع، الحقول، والتكاملات.

لم يتم استبدال هذه الأجزاء بتنفيذ CRM القديم، ولم يتم تعديل أي نظام خارج CRM.

### فروقات مؤكدة في CRM المرجعي وتم تنفيذها

1. **الحالات وترتيبها**
   - تحويل حالة الكاش المرجعية من `محتمل` إلى `لم يتم الرد` داخل تعريفات الحالة والبيانات القديمة عبر Migration آمنة.
   - تطبيق ترتيب حالات التمويل الموجود في السورس المرجعي، ووضع `لم يتم الرد` بالترتيب 210.
   - تحديث مصفوفات مؤشرات التقارير التي كانت تحتوي على `محتمل`.

2. **توحيد تاريخ التقارير**
   - التقارير تستخدم `registered_at` مع الرجوع إلى `created_at` عند غيابه.
   - بداية ونهاية اليوم محسوبة بتوقيت `Asia/Riyadh`.
   - نفس التاريخ يظهر في تفاصيل العملاء والتصدير.

3. **إعدادات مؤشرات التقارير داخل الإدارة**
   - تعريف إجمالي العملاء: كل العملاء أو حالات محددة.
   - تعريف حالات «لم يتم الاتصال».
   - تعريف حالات «مؤهل».
   - حالات بسط ومقام جودة التسويق.
   - حالات بسط ومقام جودة المبيعات.
   - منع القسمة على صفر.
   - منع مقام لا يحتوي على حالات البسط.
   - منع حفظ أسماء حالات غير معتمدة من الواجهة أو باستدعاء API مباشر.
   - ترتيب وإظهار/إخفاء كروت النتائج.
   - حفظ الإعدادات في PostgreSQL مع Audit Log.

4. **مصادر التسويق الرقمي والمباشر**
   - إضافة `report_group` إلى `core.sources` بقيم `digital` و`direct` و`other`.
   - تصنيف حاسبة التقسيط واتصال الرقم الموحد ضمن المصادر الرقمية.
   - التقارير تعتمد التصنيف المخزن، ولا تعتمد البحث النصي داخل اسم المصدر.
   - إدارة المصدر تدعم تحديد تصنيف التقرير.

5. **توحيد أرقام التقارير**
   - مصدر واحد داخل API لحساب إجمالي العملاء، لم يتم الاتصال، غير مؤهل، مؤهل، مؤجل، لم يتم الرد، تم البيع، وجودتي التسويق والمبيعات.
   - نفس الحساب يُعاد استخدامه في إجمالي التقرير ومصادر التسويق والأقسام والمناديب وخدمة العملاء.
   - «تم البيع» يعتمد حالات بسط المبيعات المحفوظة في الإدارة، ويشمل افتراضيًا حالتي البيع النهائيتين.
   - إصلاح فلتر الكول سنتر بحيث يعتمد `call_center_assigned_to` فعليًا.
   - عدم إرسال قوائم العملاء كاملة ومكررة داخل كل صف تقرير؛ تفاصيل العملاء تُحمّل عند الطلب بـServer-side pagination (100 سجل للصفحة).
   - تحديث كارت CRM في الداش بورد الموحدة من «العملاء المحتملون» إلى «لم يتم الرد» وربطه بالحالة الجديدة.
   - توحيد «جدد اليوم/هذا الأسبوع» والرسم الأسبوعي في الداش بورد الموحدة على `registered_at` بتوقيت السعودية.

6. **Excel حقيقي**
   - استبدال CSV الذي كان ينزل باسم Excel بملف `.xlsx` حقيقي.
   - دعم العربية وRTL وتجميد صف العناوين وAuto Filter.
   - حماية خلايا النص من Formula Injection.
   - لا يتم تصدير IDs أو حقول داخلية في تقارير المستخدم.
   - زر Excel في صفحة قاعدة بيانات CRM أصبح XLSX حقيقيًا ويجلب جميع نتائج الفلتر على دفعات، وليس الصفحة الحالية فقط.

7. **إضافة وتعديل العملاء اليدويين**
   - تشغيل زر التعديل فعليًا من الواجهة إلى API ثم PostgreSQL.
   - إضافة الفئة، الموديل، اللون، نوع التمويل، وتاريخ تسجيل العميل.
   - تحديث العميل الفعلي المرتبط بالطلب، وليس سجل الطلب فقط.
   - الحفاظ على المحادثة وسجل الملكية وعدم تغيير المسؤول إلا عند تغييره فعليًا؛ تعديل أي حقل آخر لا يعيد العميل إلى المندوب القديم.
   - فحص رقم الجوال بعد Normalization ومنع تحويل التعديل إلى عميل مكرر بلا موافقة.
   - تسجيل قبل/بعد في Audit Log.
   - Server-side pagination لسجلات الإضافة اليدوية بدل حد 500 ثابت.
   - مسح سجل الإضافة اليدوية أصبح Soft Delete مع Audit، فلا يُفقد السجل التاريخي أو العميل الفعلي المرتبط به.
   - صفحة قاعدة بيانات CRM وسجل عملاء التمويل وسجل وكيل صندوق الوارد تستخدم `LIMIT/OFFSET` حقيقيًا، بدون توقف العرض عند 500 سجل.

8. **مراجعة أخطاء البيانات**
   - تبويب Native داخل إدارة CRM، وليس صفحة أو نظامًا منفصلًا.
   - يفحص: تكرار الجوال، القسم/الفرع/المسؤول الناقص، تمويل بلا كول سنتر، مستخدم غير موجود أو موقوف، تعارض فرع/قسم المسؤول، حالة أو مصدر غير معروف، تاريخ ناقص، وتعارض نوع الخدمة مع القسم.
   - تنزيل شيت XLSX حقيقي يحتوي سبب الخطأ والحقل والقيمة الحالية وأعمدة التصحيح.
   - رفع الشيت وPreview قبل التنفيذ، مع قراءة خلايا التاريخ المنسقة داخل Excel وتحويلها إلى `YYYY-MM-DD`.
   - رفض الصفوف غير الصالحة، والتصحيحات المكررة، ومحاولة إعطاء نفس رقم الجوال لأكثر من عميل داخل الشيت.
   - تحميل بيانات المراجع اللازمة للتحقق بصورة مجمعة بدل N+1 Query لكل صف.
   - تنفيذ كل التعديلات داخل Transaction واحدة؛ لا يتم تنفيذ نصف الملف.
   - تسجيل كل تصحيح في `crm.lead_events` و`audit.activity_log`.

## 3) APIs المعدلة أو المضافة

| Endpoint | النوع | التغيير |
|---|---|---|
| `/api/crm/reports` | معدل | التاريخ الموحد، مؤشرات الإدارة، تصنيف المصادر، الفلاتر والحسابات الموحدة وتفاصيل العملاء المرقمة |
| `/api/crm/leads` | معدل | Pagination حقيقية، تاريخ السعودية، ودعم تصدير جميع نتائج الفلتر على دفعات |
| `/api/crm/history` | معدل | إزالة حد 500 وإضافة Pagination لسجل عملاء التمويل |
| `/api/crm/inbox-agent` | معدل | Pagination لسجل الوكيل وإجمالي السجل وعدد إجراءات اليوم بتوقيت السعودية |
| `/api/crm/manual-leads` | معدل | Pagination، الحقول الجديدة، تعديل فعلي، منع التكرار، Audit |
| `/api/crm/settings` | معدل | إعدادات المؤشرات الجديدة، تصنيف المصادر، التحقق من الحالات |
| `/api/crm/meta` | معدل | إرجاع `report_group` للمصادر |
| `/api/crm/data-review` | جديد | فحص البيانات، Preview، وتنفيذ التصحيحات داخل Transaction |

كل Endpoint يستمر في استخدام Authentication وAuthorization الخاصين بالمنصة. مراجعة البيانات وإعدادات CRM متاحتان لمديري CRM فقط.

## 4) تغييرات قاعدة البيانات

Migration الأساسية:

`database/migrations/20260720_crm_reference_v39.sql`

تشمل:

- `core.sources.report_group`.
- حقول مؤشرات التقارير الجديدة.
- حقول العميل اليدوي الجديدة وحقول Soft Delete لسجل الطلب.
- تحديث حالات الكاش وترتيب التمويل.
- ترحيل بيانات `محتمل` للكاش إلى `لم يتم الرد` دون حذف العملاء.
- رفض التشغيل المتكرر إذا كان Marker موجودًا، لمنع إعادة تطبيق Migration.
- كتابة ملخص Audit بعدد سجلات العملاء وطلبات الخدمة التي تم ترحيل حالتها.
- تحديث إعدادات التقارير القديمة.
- Indexes للتاريخ والمصدر والمندوب والكول سنتر.
- Marker: `crm-reference-v39-20260720`.

Migration لا تعمل تلقائيًا من الواجهة أو أثناء Render. يجب تنفيذها مرة واحدة على PostgreSQL قبل نشر الكود.

يوجد Rollback محافظ يزيل Indexes وMarker فقط، ولا يحذف الأعمدة الإضافية ولا يعيد بيانات الحالات تلقائيًا؛ لأن حذف أعمدة أو عكس الحالات بعد بدء الاستخدام قد يؤدي إلى فقد بيانات جديدة.

## 5) ترتيب النشر الآمن

1. أخذ Backup كامل لقاعدة PostgreSQL.
2. تشغيل:
   - `database/verification/20260720_crm_reference_v39_precheck.sql`
3. حفظ النتائج، خصوصًا أعداد الحالات والمكررات والمصادر غير المعروفة.
4. تشغيل Migration مرة واحدة:
   - `database/migrations/20260720_crm_reference_v39.sql`
5. تشغيل:
   - `database/verification/20260720_crm_reference_v39_postcheck.sql`
6. التأكد أن Assertions انتهت بدون Exception وأن Marker موجود.
7. نشر سورس المنصة 1.15.0.
8. اختبار Admin ومدير المبيعات ومدير الفرع والمندوب والكول سنتر.
9. مقارنة نفس الفترة والفلاتر بين الداش بورد والتقارير والتصدير والداش بورد الموحدة.

## 6) الحماية من فقدان البيانات

- لا يوجد `DROP TABLE` أو `TRUNCATE` أو Seed يمسح بيانات.
- هذا التحديث وMigration الخاصة به لا يحذفان عميلًا أو محادثة أو سجل حالة أو مستخدمًا.
- لا توجد إعادة توزيع جماعية للعملاء الحاليين.
- تحويل الحالة القديمة محدد لعملاء الكاش فقط.
- التعديلات الجماعية من شيت مراجعة البيانات تستخدم Preview ثم Transaction واحدة.
- Migration منفصلة وقابلة للمراجعة، وليست جزءًا مخفيًا من تشغيل الواجهة.

## 7) الأنظمة غير المتأثرة

لم يتم تعديل ملفات أو منطق العمليات أو التراكينج أو التسويق. تعديلات الملفات المشتركة اقتصرت على تسجيل Route الخاص بـCRM ورقم إصدار API، وتوسعة Schema CRM/المصادر المشتركة بالحقول التي يحتاجها CRM فقط.

## 8) الاختبارات المنفذة

التفاصيل الكاملة موجودة في:

`docs/CRM-REFERENCE-V39-TEST-RESULTS.txt`

تم تنفيذ فحوص الاستيرادات، وحدود الموديولات، واختبارات CRM الحالية، واختبارات Regression للعمليات والتراكينج، وفحص Reference v39 الجديد، وفحص syntax/TypeScript موجه للملفات المعدلة، والتحقق من ملف XLSX باستخدام قارئ مستقل.

## 9) حدود التحقق في بيئة التسليم

- لم يتم الاتصال بقاعدة الإنتاج، لذلك لم يتم تشغيل Migration أو فحص أعداد البيانات الفعلية؛ تم تسليم Scripts precheck/postcheck لهذا الغرض.
- تعذر تنفيذ `pnpm install` وProduction Build كامل لأن بيئة العمل لا تملك اتصالًا بالشبكة ولا تحتوي Dependencies المشروع الأصلية. لم يتم إخفاء ذلك أو تعطيل TypeScript؛ تم بدلًا منه تشغيل جميع الفحوص المتاحة واختبارات TypeScript موجهة باستخدام تعريفات مؤقتة لم تدخل ملف التسليم.
- يجب تشغيل `pnpm install --frozen-lockfile && pnpm run build` في بيئة CI/النشر المتصلة قبل اعتماد النشر النهائي.

## 10) شروط القبول التي تم التركيز عليها

- التنفيذ داخل المنصة الأصلية بدون Firebase أو Copy/Paste من CRM القديم.
- لا Patch فوق Patch ولا ملفات بديلة تعمل بالتوازي.
- إعدادات التقارير في PostgreSQL.
- Excel حقيقي في التقارير وقاعدة بيانات CRM وشيت مراجعة البيانات.
- تعديل العميل يعمل فعليًا.
- مراجعة أخطاء البيانات تعمل من الفحص إلى Preview ثم Transaction وAudit.
- التصنيف الرقمي/المباشر مخزن في قاعدة البيانات.
- عدم تغيير الأنظمة الأخرى أو تصميم المنصة.
