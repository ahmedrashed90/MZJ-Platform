## سبب خطأ requires_note

قاعدة البيانات الموجودة كانت تحتوي على جدول `operations.vehicle_statuses` من محاولة سابقة، لكن بدون العمود `requires_note`، بينما الكود حاول استخدامه أثناء Seed الحالات. المهاجرة الجديدة تنفذ بالترتيب الآتي داخل Transaction واحدة:

1. إنشاء الجدول إن لم يكن موجودًا.
2. إضافة `requires_note` وباقي الأعمدة الناقصة باستخدام `ADD COLUMN IF NOT EXISTS`.
3. تنفيذ Seed/Upsert للحالات بعد اكتمال الأعمدة.
4. تسجيل Migration Key بعد نجاح كل الخطوات.

تم اختبار الترقية على Schema قديمة مماثلة بعد حذف العمود عمدًا، ثم تشغيل المهاجرة مرتين.

## حذف طلب التراكينج

الحذف موجود داخل تفاصيل طلب التراكينج ويحتاج `tracking.orders.delete` أو مدير النظام. العملية تنفذ في Transaction واحدة:

- قفل الطلب.
- أخذ Snapshot للطلب والسيارات والمراحل والأحداث والرسائل.
- حفظ سجل الحذف الدائم.
- تسجيل Source Fingerprint للنسخة القديمة فقط.
- حذف الطلب وروابطه من التراكينج.
- الإبقاء على السيارة داخل مخزون العمليات.
- قبول طلب جديد مستقبلًا بنفس رقم الطلب إذا جاء من Source Identity جديدة.

نافذة التأكيد تُعرض عبر React Portal بطبقة `z-index: 10000` أعلى من نافذة التفاصيل.

   `database/migrations/015_operations_tracking_clean_rebuild.sql`
   على Staging أولًا.
4. التحقق من وجود العمود:

```sql
select column_name
from information_schema.columns
where table_schema='operations'
  and table_name='vehicle_statuses'
  and column_name='requires_note';
```

5. اختبار المخزون، الحركة، طلب نقل، الموافقات، وحذف طلب Tracking تجريبي قبل النشر على Main.

## Rollback

المهاجرة غير تدميرية ولا تحذف السيارات أو التاريخ. للعودة التشغيلية:

- إعادة نشر الإصدار السابق.
- عند الحاجة إلى رجوع كامل للبيانات، استعادة Backup قاعدة البيانات المأخوذ قبل النشر.
- لا تنفذ DROP يدويًا للجداول أو الأعمدة الجديدة على Main.
