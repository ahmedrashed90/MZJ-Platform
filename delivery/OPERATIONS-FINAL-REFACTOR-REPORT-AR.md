# تقرير تسليم تعديلات نظام العمليات — MZJ Platform v1.13.3

التاريخ: 2026-07-19

## نطاق التنفيذ

تم العمل على آخر سورس كامل مرفوع، مع تعديل الأجزاء المحددة داخل نظام العمليات فقط. لم يتم تغيير منطق CRM أو التسويق أو التراكينج أو تسجيل الدخول. تم الحفاظ على بيانات طلبات التصوير التاريخية في قاعدة البيانات، مع فصلها عن واجهة ومسار طلبات النقل الحالية.

## ما تم تنفيذه

### 1. تبويبات العمليات
- توحيد شريط التبويبات في Component واحد.
- عرض ثمانية تبويبات فقط، وتغيير الاسم إلى «طلبات النقل».
- توسيط التبويبات على Desktop، ودعم التمرير الأفقي على الشاشات الصغيرة، وتحريك الشريط تلقائيًا إلى التبويب النشط.

### 2. مخزون السيارات
- تثبيت الفلاتر ورأس الجدول أثناء التمرير.
- إضافة شريط تمرير أفقي ثابت ومتزامن مع الجدول.
- إضافة تغيير عرض الأعمدة بالسحب، وDouble Click للـAuto Fit، وإعادة ضبط المقاسات.
- حفظ مقاسات الأعمدة كتفضيل واجهة فقط في Local Storage.
- استمرار الفلترة والتقسيم من السيرفر، وتصدير جميع النتائج المطابقة مع VIN كنص.

### 3. مسح السيارة النهائي
- إضافة زر «مسح السيارة» داخل إجراءات السيارة.
- لا يظهر إلا لـsystem_admin أو لصلاحية operations.vehicle.delete.
- نافذة تأكيد تعرض VIN واسم السيارة وتطلب سببًا وتأكيدًا صريحًا.
- فحص Server-side داخل Transaction لكل العلاقات: الحركات، الموافقات، سجل الموافقات، طلبات النقل، الأرشيف، Tracking، التشيك الحالي والتاريخي، ملاحظات الحالات، والأحداث التشغيلية.
- رفض المسح برسالة VEHICLE_HAS_HISTORY وطلب الأرشفة عند وجود أي تاريخ.
- تحويل أي Foreign Key غير متوقع إلى رفض منظم بدل عرض خطأ قاعدة بيانات خام.
- تسجيل Audit Trail ببيانات السيارة قبل المسح، المستخدم، السبب، والتوقيت.
- تحديث المخزون والداش بورد تلقائيًا بعد النجاح دون Refresh يدوي.

### 4. استيراد Excel
- تنفيذ ثلاثة أوضاع فقط: الاستبدال الكامل، الإضافة فوق الحالي، والتحديث من الشيت.
- إضافة مرحلة معاينة وتقرير صفّي قبل التنفيذ وبعده.
- إضافة Import Batch وRequest Key لمنع Double Submit.
- الاستبدال الكامل آمن وغير تدميري: يعطل اللقطة التشغيلية القديمة بدل الحذف الفيزيائي ويحافظ على التاريخ والعلاقات.
- إضافة فوق الحالي تضيف VINs الجديدة فقط.
- تحديث من الشيت يعدل السيارات الموجودة فقط ويحافظ على القيم القديمة عند الحقول الفارغة.
- تغيير المكان أو الحالة في وضع التحديث مرفوض؛ ويطلب استخدام تبويب الحركة حتى لا يتم تجاوز سجل الحركة أو دورة الموافقات.
- VIN يُعامل كنص، مع منع Scientific Notation والتكرار والفراغ.
- صلاحية الاستبدال الكامل منفصلة: operations.vehicles.replace أو system_admin.

### 5. الحركة
- عرض بيانات كاملة ومستقلة لكل سيارة مختارة.
- منع تكرار السيارة وعدم مسح بيانات باقي السيارات عند إزالة واحدة.
- جلب الحالات الفعلية من PostgreSQL/API بدل Hardcode داخل الواجهة.
- إضافة Preview جماعية قبل التنفيذ مع سبب السماح أو المنع لكل سيارة.
- تنفيذ Atomic لعدة سيارات مع Batch ID وRequest Key.
- «مباع تحت التسليم» ينشئ دورة موافقات مالية وإدارية بحالة لم يتم، ويحدث تبويب وكارت الموافقات.
- «مباع تم التسليم» لا يُقبل قبل المرور بالحالة السابقة واكتمال الموافقتين لنفس الدورة.

### 6. طلبات النقل
- الصفحة أصبحت مخصصة للنقل فقط.
- التبويبات الداخلية: إنشاء طلب، متابعة الطلبات، الطلبات المكتملة.
- إزالة اختيار نوع الطلب، طلبات التصوير، جميع الطلبات، وسبب الطلب من الفلو الجديد.
- إنشاء الطلب يدعم سيارة أو عدة سيارات مع بيانات ومراجعة كاملة.
- منع السيارة المكررة، والخارجة عن الصلاحية، والمؤرشفة، وغير النشطة، والمرتبطة بطلب نقل نشط.
- تنفيذ إنشاء الطلب والعناصر والأحداث داخل Transaction.
- المراحل المتتابعة محمية من التخطي والتكرار، واستلام السيارة يحدث المكان والحركة والمرحلة معًا.
- كارت الداش بورد يعتمد نفس Service وQuery ويحسب طلبات النقل فقط.

### 7. الموافقات ومصادر الحقيقة
- توحيد مصدر الموافقات بين الداش بورد، تبويب الموافقات، تفاصيل السيارة، والحركة.
- التحقق من الصلاحيات داخل API للموافقة المالية والإدارية والتراجع.
- تحديث الواجهات والداش بورد عبر invalidation/event منظم بدل Polling ثقيل.

### 8. Backend وDatabase
- إضافة Structured Errors للأخطاء التشغيلية الجديدة.
- إضافة Migration غير تدميرية لحقول اللقطة التشغيلية، Import Batches، Request Keys، والصلاحيات الجديدة.
- لم يتم Drop لأي جدول أو حذف أي بيانات تاريخية.

## الأسباب الجذرية التي عولجت

1. صفحة الطلبات كانت تجمع النقل والتصوير في Route وService وواجهة واحدة، مما تسبب في خلط المسؤوليات والعدادات.
2. الاستيراد كان مسار Upsert واحدًا بلا فصل واضح بين الإضافة والتحديث والاستبدال، وبلا Preview أو Import Batch.
3. قائمة حالات الحركة لم تكن مبنية بالكامل على مصدر الحالات الفعلي والانتقالات Server-side.
4. الجدول لم يكن يملك طبقة مشتركة لإدارة قياسات الأعمدة أو مزامنة التمرير.
5. تحديث الداش بورد بعد التغييرات كان يعتمد على إعادة تحميل يدوي بدل حدث موحد.
6. لم يكن هناك إجراء مسح فيزيائي آمن يفرق بين سيارة بلا تاريخ وسيارة مرتبطة بعلاقات تشغيلية.

## المكونات والمسارات المكررة

لم يتم إنشاء Route جديد بجانب Route قديم يؤدي نفس الوظيفة. تمت إعادة بناء الصفحات والخدمات المتأثرة داخل ملفاتها الحالية. لم يُحذف أي ملف Runtime كامل، لكن تمت إزالة منطق التصوير من صفحة ومسار إنشاء طلبات النقل، مع الإبقاء على الجداول والبيانات التاريخية.

الخطوات المطلوبة:

2. تشغيل الـMigration على نسخة Test مطابقة لـmain.
3. التحقق من الصلاحيتين operations.vehicle.delete وoperations.vehicles.replace؛ لا تمنحان تلقائيًا لمدير العمليات، بينما system_admin يحصل عليهما.
4. تشغيل اختبارات القبول على Test.
5. تشغيل الـMigration على main بعد نجاح التحقق.

## نتائج الاختبارات

نجحت اختبارات القبول الثابتة واختبار Syntax/isolated transpilation لـ110 ملف TypeScript/TSX. التفاصيل في `delivery/TEST-REPORT.txt`.

لم يتم تنفيذ Production Build أو tsc -b في بيئة العمل الحالية لأن registry.npmjs.org لم يكن متاحًا، ولم تكن الحزم مثبتة محليًا. كما لم يتم تشغيل Migration على قاعدة main أو تنفيذ اختبار Browser E2E على Vercel. يجب تنفيذ هذه الخطوات قبل اعتماد النشر النهائي.

## اختبارات القبول اليدوية المطلوبة قبل main

- تجربة الصلاحيات على system_admin ومستخدم يملك operations.vehicle.delete ومستخدم لا يملكها.
- مسح سيارة جديدة بلا أي علاقة، والتحقق من Audit Trail وتحديث الداش بورد.
- محاولة مسح سيارة لها كل نوع من العلاقات، والتأكد من رفض المسح واقتراح الأرشفة.
- تجربة أوضاع الاستيراد الثلاثة بملف صحيح وملف به VIN مكرر وVIN بصيغة Scientific Notation.
- التأكد من أن الاستبدال لا يحذف حركات أو موافقات أو طلبات أو Tracking أو أرشيف.
- تجربة حركة متعددة السيارات وفشل سيارة واحدة للتحقق من Rollback الكامل.
- تجربة «مباع تحت التسليم» ثم الموافقتين ثم «مباع تم التسليم».
- إنشاء طلب نقل متعدد السيارات وتنفيذ المراحل بالترتيب ثم مراجعة Timeline والحركة الناتجة.
- مقارنة كروت الداش بورد مع صفحات الموافقات وطلبات النقل بعد كل إجراء.
